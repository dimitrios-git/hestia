# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

Personal dotfiles for a Debian-based Linux system running the **Sway (Wayland)** desktop. No build system, no tests, no linting — changes take effect by editing the files directly (the active ones are symlinked into `~` / `~/.config`). Reproducing the machine from scratch is driven by an **Ansible bootstrap** (`bootstrap/`, a dozen roles — see *Bootstrap & system reproduction*); the ordered fresh-install runbook is `docs/install-runbook.md`.

The repo is growing from *one person's dotfiles* toward a **reproducible Debian spin** with multiple principals and `/etc` system configs; the target layered structure is `docs/repo-structure-design.md` (phased, not yet migrated).

The configs are converging on one unified theme: dark ground `#1a1a1a` (lifted from near-black `#0a0a0a` in 2026-07, palette 0.5.0), accent **red `#d7005f`** (ANSI color01), and a saturated 16-colour palette derived from the **`wildcharm`** vim colorscheme (the accent was Debian red `#ce0056` until 2026-06; dropped so the theme is uniform with the scheme it is built on). Vim/Neovim use the **`hestia`** colorscheme (`user/vim/colors/hestia.vim`) — since 2026-07 a self-contained scheme **GENERATED by `render.py`** from `palette.yml` (a thin wrapper over built-in `wildcharm` until then); inspired by `wildcharm` (the palette was derived from it) but now owned by hestia and identical in Vim and Neovim. The palette is now defined once in **`themes/hestia/palette.yml`** (the single source of truth — semantic roles, the 16 ANSI colours, and a versioned **`syntax:`** role table for code highlighting whose canonical mapping is wildcharm.vim), and the **per-app theming process** (+ a themed/not-yet status table) is **`docs/theming.md`**. Keep new theming consistent with this — and when theming an app, follow that process. Since M7 (palette 0.6.0) the desktop can also run the **light variant** — ground `#f5f5f5`, same accent — switched by the bootstrap's **`theme_variant: dark|light`** (host_vars; setup.sh asks; flip = re-run `--tags dotfiles,sway_session` + **re-login**); the dark/light system and cross-platform syntax coherence (vim / bat / Shiki / VS Code) — layer model, decision log, milestones, inconsistency backlog — are governed by **`docs/theme-roadmap.md`**; syntax consumers render from the `syntax:` table, never ad hoc. The TextMate-family artifacts (bat's `.tmTheme`, the web Shiki pair in `themes/hestia/dist/shiki/`, the VS Code themes in `user/vscode/hestia/`) **and the per-app colour files** (the `user/{kitty,sway,mako}/theme-{dark,light}.conf`, `user/{waybar,wofi}/theme-{dark,light}.css` and `user/zathura/theme-{dark,light}` fragments included/`@import`ed by the main configs; the `user/{swaylock,swaynag}/config-{dark,light}` whole-file pairs; and the `user/vifm/colors/hestia-*.vifm` / `user/bat/themes/hestia-*.tmTheme` / `user/glow/hestia-*.json` theme pairs — in every case the `theme_variant` one is symlinked to a variant-neutral dest — plus the single-file `user/thunderbird/user{Chrome,Content}.css`, which carry BOTH variants in one file via CSS `light-dark()`, no pair/symlink; and the Vim/Neovim colorscheme `user/vim/colors/hestia.vim`, one symlinked file carrying both `&background` branches) are **GENERATED** by **`themes/hestia/render.py`** from `palette.yml` + **`scopes.yml`** (the shared role→scope map) — never hand-edit them; edit the sources, re-render (`render.py`), and use `render.py --check` to detect drift.

## Running as the `claude` agent user

Claude Code runs here as a dedicated, unprivileged **`claude`** Linux user (own `0700` home `/home/claude`, member of the **`devshare`** group, **not** in sudo) — a kernel-enforced trust boundary, not a behavioural one. Full design: `docs/claude-user-design.md` (implemented 2026-06-20); day-to-day workflow: `docs/working-with-claude.md`. What matters when working in this repo:

- **This checkout (`/srv/devshare/hestia`) is claude's own clone** in the shared `/srv/devshare` tree — claude's *workspace*, **not** the live system. The dotfiles are symlinked from **dimitrios's** clone at **`~/Development/hestia`**, which is the deployment source. Edits made here reach the running system only when **dimitrios pulls** after reviewing a pushed branch — so an edit here is **not** live. Never treat `/srv/devshare/hestia` as the deployment source. (No `/home/dimitrios/...` literals remain in any deployed config — every host-specific path resolves from `target_home`/`repo_root`/`$HOME`/host_vars at deploy time. Path generalisation is **complete**, `docs/repo-structure-design.md` §5.)
- **Collaboration is git-mediated, two principals.** claude commits and signs as **itself**, pushes a topic branch, and opens a **pull request** as the GitHub bot **`dimitrios-claude`** (`gh pr create`; commits show **Verified**); dimitrios reviews, **merge-commits** (preserves the signed commits; via `gh pr merge --admin`), and pulls to deploy. claude **never** self-merges — and *cannot*: a `main` ruleset (*Restrict updates*, admin-only bypass) blocks the agent from pushing to or merging `main` server-side, so its only route is opening a PR. The concrete loop (and the `gh`/PAT auth) is in `docs/working-with-claude.md`.
- **claude's identity:** own SSH key `~/.ssh/id_claude`, own **passwordless** GPG key `4AA9DD310356AD0E`, git author `Claude (dimitrios's agent) <claude@charalampidis.pro>`.
- **Signing is frictionless for claude** — the passwordless key needs **no pinentry, no agent cache, no `gpg-unlock`, and not `gpg-wrapper.sh`**. All that machinery in *Secrets & commit signing* below is **dimitrios's**; it does not apply when running as claude.
- **Isolation:** claude **cannot** read dimitrios's home, `~/.ssh`, `~/.gnupg`, or `~/.bash_secrets` (different UID; `0750`/`0700`/`0600`). Confirm any time: `ls ~dimitrios` → Permission denied.
- **`user/bin/claude-access`** (`grant|revoke|list`, symlinked to `~/.local/bin/`) is the escape hatch to give claude ACL access to a path **outside** `/srv/devshare` (e.g. a project in the human's home), with a tracked registry. Prefer the clone-and-sync model; reach for it only when in-place editing is truly needed.
- **claude can install its own user-level tooling.** Because `claude` is a separate user with its own `/home/claude`, it can freely install whatever a task needs **into its own home** — npm/pip `--user`, cargo, a downloaded binary into `~/.local/bin`, etc. — without `sudo` and without touching dimitrios's environment. (System-wide `apt` installs still need the human; those stay dimitrios's job.)

## File layout and naming convention

Top level: **`user/`** (the human's home-deployed configs — one subdir per tool), **`system/`** (root-owned `/etc` configs), **`bootstrap/`** (the Ansible installer), **`themes/`** (the theme single-source-of-truth — `hestia/palette.yml`; see *Theming* / `docs/theming.md`), **`docs/`** (maintainer design docs/runbooks), **`showcase/`** (READER-facing docs: per-category evaluation-trace chapters + screenshots — index `showcase/README.md`; each concluded evaluation ships its chapter in/beside the verdict PR; plain markdown + relative `img/` paths so chapters syndicate to the stoa charalampidis app unchanged). The `user/` grouping is a flat namespace — *not* `users/<name>/`: per-user values are parameterised (host_vars/templating, §6), so a username sub-level would be empty ceremony (see `docs/repo-structure-design.md` §7).

Active `user/` configs are either:
- **Dot-prefixed** in the repo, symlinked into `$HOME` — `user/vim/.vimrc` → `~/.vimrc`, `user/bash/.bashrc` → `~/.bashrc`, `user/git/.gitignore_global` → `~/.gitignore_global`.
- **Plain files under the app dir**, symlinked into `~/.config/<app>/` — sway, waybar, mako, wofi, cava, cmus, kitty, imv, vifm, glow (`hestia.json`), nvim, xdg-desktop-portal.
- **Rendered (not symlinked)** — a small **path-generalised / identity-bearing** set rendered from `.j2` by the bootstrap (`user/git/.gitconfig`, `user/glow/glow.yml`, `user/cmus/rc`, and the two credential scripts `credential-unlock.sh` + `keyring-ssh-askpass.sh`); see *Rendered (templated) configs* below.
- **`user/bin/`** — small helper scripts symlinked into `~/.local/bin/` (currently `claude-access`; see *Running as the `claude` agent user*).
- **`system/`** — layer-(a) **system configs** deployed by **copy**/template to root-owned locations (**not** symlinked), driven by Ansible roles: `system/samba/smb.conf` → `/etc/` (`samba` role); `system/sway-session/{start-sway → /usr/local/bin/, greetd-config.toml → /etc/greetd/config.toml}` (`sway_session` role). Each `system/` subdir keeps a `README.md` runbook.
- **`bootstrap/`**, **`docs/`** — the Ansible installer + the design docs/runbooks (cross-cutting, not a per-tool config).

Two `/srv` trees live outside the repo but are part of the setup: **`/srv/devshare`** (the human↔`claude` shared dev tree — group `devshare`, setgid + default ACLs; this clone lives here) and **`/srv/smbshare`** (the Samba share — group `smbshare`; `docs/file-sharing-design.md`).

## Secrets & commit signing

> This section describes **dimitrios's** interactive setup (the passphrase-protected key + the pinentry-wedge machinery built around it). When running as the **`claude`** user, signing is **passwordless** and none of the wrapper / cache / `gpg-unlock` flow applies — see *Running as the `claude` agent user*.

- **Secrets** (API tokens, etc.) live in **`~/.bash_secrets`** — untracked, mode 600, sourced at the end of `user/bash/.bashrc`. Never commit secret values; a bootstrap script must not carry this file.
- **Commits are GPG-signed** (the signing key id is an **installer input** — `git_signingkey` in host_vars, rendered into both `user/git/.gitconfig.j2` and `credential-unlock.sh`; empty → `gpgsign = false`). Signing is routed through **`user/git/gpg-wrapper.sh`** (set as `gpg.program` in `user/git/.gitconfig.j2`, templated from `{{ repo_root }}` so it follows the clone — no manual update on move). In a normal terminal it execs `gpg` untouched, using **`pinentry-tty`** (configured in **`user/gnupg/gpg-agent.conf`**, symlinked to `~/.gnupg/`) with a **~session-length cache TTL** (`34560000`) — because login auto-unlock (below) warms the agent at login, so the security boundary is the unlocked session + screen lock, not the TTL; GPG now matches ssh-agent's persist-until-logout.
- **Why the wrapper:** inside Claude Code (`$CLAUDECODE` set) gpg-agent would launch pinentry on `GPG_TTY` — which is Claude's own terminal (`/dev/pts/0`), **not** the calling process's stdin — and **seize it** (you can't type the passphrase or control the prompt; `pinentry-tty` does *not* avoid this, the tty takeover is the problem). So under `$CLAUDECODE` the wrapper adds `--pinentry-mode error`: a **warm** cache signs silently, a **cold** cache fails fast (`gpg: signing failed: No pinentry`, exit 2) instead of wedging. When a harness commit fails this way, run **`gpg-unlock`** (a `user/bash/.bashrc` function — signs a throwaway message so gpg-agent prompts once via pinentry-tty; reads the key from git config, so no hash to remember; refuses to run under `$CLAUDECODE`) in a normal terminal, then the harness can sign again within the cache window.
- **Bootstrap / pinentry default:** the tracked `gpg-agent.conf` forces `pinentry-tty` per-agent, but the *system* `pinentry` alternative defaults to **`pinentry-curses`** (leaks mouse-tracking escape codes). Pin it belt-and-suspenders with `sudo update-alternatives --set pinentry /usr/bin/pinentry-tty`. Avoid `pinentry-curses` and `pinentry-gnome3` (the latter falls back to curses outside GNOME).
- **Login auto-unlock (SSH + GPG):** `pam_gnome_keyring` unlocks the gnome-keyring login keyring at login; the Sway-`exec` hook **`user/gnupg/credential-unlock.sh`** then loads the SSH key (via an SSH_ASKPASS helper reading the keyring) and warms gpg-agent (loopback sign), so **no per-boot `ssh-add`/`gpg-unlock`** — including for the harness. Passphrases live in the keyring (`secret-tool store … autounlock ssh/gpg …`). Full story incl. the gnome-keyring launcher-untangle needed to make PAM the sole unlocker: `docs/credential-autounlock-design.md` §10. The wrapper + `gpg-unlock` remain as fail-clean fallbacks for a cold cache.

## Bootstrap & system reproduction (`bootstrap/`)

A fresh machine is reproduced by an **Ansible** playbook (engine decision: `docs/repo-structure-design.md`). The one-shot entry point is **`bootstrap/setup.sh`** — installs Ansible, interactively gathers host answers (the §6 feature toggles + `samba_lan_subnet` + `cmus_music_dir` + `theme_variant` dark|light, with auto-detected defaults) into the untracked `host_vars/localhost.yml`, then runs the playbook (re-runnable; own flags `--no-backup`/`--yes`/`-h`/`--help`, everything else passes through — and `--check` is a true dry-run: answers go to a temp file, real `host_vars` untouched, ansible only previews). It **captures the sudo password up front with its own retry loop** (validating each attempt via `sudo -S -v`, 3 tries) and hands it to ansible via `--become-password-file` pointing at a **tmpfs file** (`$XDG_RUNTIME_DIR`/`/dev/shm` — RAM-backed, `0600`, removed right after the run), instead of ansible's single-shot `--ask-become-pass` where one typo aborts the whole play. (Two things that *don't* work, learned the hard way: a cached `sudo -v` timestamp — sudo's `tty_tickets` keys it to the shell's tty but ansible's become runs on a different tty; and process-substitution `<(…)` for the password file — ansible re-opens the path by name and `/dev/fd/N` resolves to an unopenable `pipe:[inode]`.) The NOPASSWD probe runs `sudo -k` first so a warm cache can't masquerade as passwordless. **`--yes`/`-y`** skips the questionnaire and reuses the saved `host_vars` as-is (resume a failed run). **It is destructive on a fresh `$HOME`** (the `dotfiles` role force-replaces existing dotfiles) — so a *first run* (no host_vars yet) shows a ⚠️ replace notice (in dry-run too — it's a simulation) and on a real apply requires typing `yes`, and **on the first deploy only** (marker-tracked at `~/.local/state/hestia/.dotfiles-backed-up`) the role copies any pre-existing real config to `<file>.bak` next to it — one `.bak` per file, ever, no accumulation. Disable with `--no-backup` (= `-e dotfiles_backup=false`); then replaced configs aren't saved. The ordered start-to-finish narrative — which role when, with the interactive/external manual steps interleaved — is **`docs/install-runbook.md`**; the per-role reference is `bootstrap/README.md`.

- **Single source of truth: `bootstrap/group_vars/all.yml`** — the manifest listing every dotfile symlink (`dotfile_links`) and the apt package set (`apt_packages`). The "Active symlinks" table above is **generated** from it by `bootstrap/gen-symlink-table.py`; run that after editing `dotfile_links` rather than hand-editing the table.
- **Per-host personal values** (e.g. the Samba `samba_lan_subnet`) live in an **untracked** `bootstrap/host_vars/<inventory_host>.yml` (template: the committed `…example`), kept out of this public repo by `.gitignore`. Ansible auto-loads it; role defaults carry RFC 5737 placeholders so a checkout is runnable without it.
- **Roles** (`bootstrap/site.yml`, sliceable by `--tags`): `packages` (apt — and, after install, it disables iperf3's bundled server daemon: iperf3 ships as a client-side throughput diagnostic, but its `iperf3.service` would otherwise listen on tcp/5201, so the role keeps the unit disabled+stopped), `dotfiles` (symlink the manifest + render `templated_configs` into `$HOME`, no root), `fonts` (the Nerd Fonts the configs reference — `Lilex`/`BigBlueTermPlus` — fetched from the pinned nerd-fonts release into `~/.local/share/fonts`, no root), `tailscale` (installs the Tailscale mesh VPN from its own signed apt repo — the share's transport + general remote reach; `tailscale up` auth stays manual), `firefoxpwa` (the PWAsForFirefox native connector from its vendor packagecloud repo — the `FFPWA-*` PWA windows the sway config themes/idle-inhibits; the in-Firefox extension + per-site installs stay manual, and the role prints that reminder), `chrome` (opt-in `enable_chrome`, **default `false`** — Google Chrome `google-chrome-stable` from Google's own signed apt repo, since Chrome is not in Debian apt (that ships `chromium`, the headless-dev build — this is the proprietary Google desktop browser); ## Tool configurations

### Vim (`user/vim/.vimrc`)
Uses **vim-plug** for plugins. To install plug.vim on a fresh system:
```sh
curl -fLo ~/.vim/autoload/plug.vim --create-dirs \
     https://raw.githubusercontent.com/junegunn/vim-plug/master/plug.vim
```
Then open Vim and run `:PlugInstall`. CoC extensions install automatically via `g:coc_global_extensions`. The config targets both Vim and Neovim — some plugins are conditional on `has('nvim')`.

Key plugins: `coc.nvim` (LSP/completion), `ctrlp.vim` (fuzzy finder), `vim-fugitive` (git), `copilot.vim` (requires Vim ≥ 9.0.0185), NERDTree, `vim-devicons` (Nerd Font filetype/folder glyphs in NERDTree/CtrlP). Theme: the `hestia` colorscheme (`user/vim/colors/hestia.vim`, GENERATED from `palette.yml` by `render.py` — self-contained, no longer a `wildcharm` wrapper); in the `theme_variant` the symlinked `~/.vim/hestia-background.vim` selector sets (`set background=dark|light`, sourced by `.vimrc` before `colo hestia`; falls back dark). **`vim-devicons` must be the last `Plug`** (it patches the plugins it decorates at load time) and needs `set encoding=UTF-8` + a Nerd Font in the terminal (we have both — Lilex Nerd Font via kitty). Icon colours mirror the GTK file manager: **folders hestia red `#d7005f`, every other icon neutral grey** (`#9e9e9e` on dark / `#626262` on light — per `&background`; icon only, names keep their normal colour). devicons injects the glyph into NERDTree's `[…]` flag region, so all icons inherit **`NERDTreeFlags`** (which links to `Number` → a hestia red — the source of the stray red on file icons); the `FileType nerdtree` autocmd `s:HestiaNerdTreeIcons` recolours `NERDTreeFlags` to grey (`ctermfg=247`) for the default, then a syntax match (`hestiaDevIconFolder`, built from devicons' own folder-symbol vars — no hardcoded codepoint) overrides the folder glyph back to red (`ctermfg=161`, wins via `containedin=ALL`). cterm cells are exact, so colours hold with or without `termguicolors`. NERDTree colours via highlight groups, **not** `LS_COLORS`/dircolors. Full colour-by-filetype would need a companion plugin (`vim-nerdtree-syntax-highlight`).

**Markdown viewing** — three layers:
- `:Glow` (or `<leader>md` = `\md`, markdown buffers only) renders the current file in **glow** as a full-screen pager in the **same terminal** (suspends Vim until `q`, like vifm), via `:!glow -p %`. Works in **both Vim and Neovim** — this is the plain-Vim path. Calls `GlowPreview()`; glow gets a real TTY so theming comes from `~/.config/glow/glow.yml` (see Glow section).
- `render-markdown.nvim` — live **in-buffer** rendering, **Neovim only** (needs treesitter/Lua, absent in plain Vim). Set up in a `has('nvim')` Lua block; headings themed to `#d7005f`. Dormant until `:PlugInstall`.
- `markdown-preview.nvim` — browser-tab live preview, **Neovim only**; kept as the escape hatch for diagrams/math.

### Neovim (`user/nvim/init.vim`, `user/nvim/lua/trees.lua`)
Neovim **shares the Vim config** rather than duplicating it. `init.vim` prepends `~/.vim` to `runtimepath` (so `plug.vim` in `~/.vim/autoload` and plugins in `~/.vim/plugged` are reused) and then `source`s `~/.vimrc`. The `has('nvim')` branches in `.vimrc` then activate the Neovim-only plugins (`render-markdown.nvim`, `markdown-preview.nvim`) and `lua require('trees')` (treesitter config in `user/nvim/lua/trees.lua`).

Fresh-install steps (Debian trixie ships **0.10.4**, recent enough — `apt install neovim`, package is `neovim`, binary is `nvim`):
1. `nvim` → `:PlugInstall` (fetches the nvim-only plugins into the shared `~/.vim/plugged`; `markdown-preview.nvim` runs `npm install`, so Node/NVM must be loaded).
2. Treesitter parsers compile on first run — needs a **C compiler** (`gcc`). `:TSUpdate` if any fail.
3. Open a `.md` file: `render-markdown.nvim` renders in-buffer (headings themed `#d7005f`); `:Glow` still works as the shared full-screen pager.

### Bash (`user/bash/.bashrc`)
- Vi mode (`set -o vi`); git-aware prompt (cwd, branch, staged/unstaged/untracked counts, exit-status colour). The prompt is **identity-adaptive** from one shared rc: a normal login user gets `path branch $` (no username), but **root** prepends a **red** `\u` and the **`claude`** agent an **orange** (`38;5;208`) `\u` → `user path branch $`. Same file, coloured by `$EUID`/`$USER` — so `sudo su` shows red because **root sources this same `.bashrc`**, not the caller's (a "go red when root" test in the *caller's* prompt never fires once you're elevated). Needs the rc deployed where it's sourced: it's symlinked into dimitrios's home today; `claude`'s home gets it via a stopgap symlink (the bootstrap will own this once claude's env is automated — `tmp.txt`), and `/root/.bashrc` only goes red once the rc reaches root (a manual/optional `ln -s` — root's home isn't managed yet).
- Loads **NVM** on startup. Sets `GPG_TTY=$(tty)` (needed for terminal GPG signing), exports `EDITOR`/`VISUAL=vim`, and prepends `~/.local/bin` to `PATH` (guarded against duplicate entries). (The system-wide `editor` **alternative** is also pointed at `vim.basic` by the `packages` role — Debian defaults it to nano; git uses `core.editor=vim` regardless.)
- **`gpg-unlock`** function: caches the GPG signing passphrase for the session (prompts once via pinentry-tty), so a cold cache stops failing harness commits. Reads the key from git config; refuses to run under `$CLAUDECODE`. See the signing section.
- **SSH agent:** exports `SSH_AUTH_SOCK="$XDG_RUNTIME_DIR/openssh_agent"`. The agent itself is **Debian's shipped socket-activated user unit** (`/usr/lib/systemd/user/ssh-agent.socket`, already `enable`d — auto-starts on login); we deliberately use that rather than a custom unit. greetd doesn't source `.bashrc`, so this export covers interactive shells (terminals, Ansible). The SSH key is now **auto-loaded at login** by the credential-unlock hook (see the signing section), reading the passphrase from the gnome-keyring login keyring — so no per-boot `ssh-add`. It persists the whole session. (Fallback if the hook ever doesn't run: `ssh-add ~/.ssh/<your key>`, i.e. the `ssh_key_file` host_var.)
- **ssh TERM fix:** under kitty (`TERM=xterm-kitty`), `ssh` is aliased to advertise `TERM=xterm-256color` — servers rarely have the `xterm-kitty` terminfo, so without this backspace/arrows garble on remote shells. Plain `ssh` keeps working keys **and** the server's login MOTD; use `kitty +kitten ssh <host>` explicitly for full kitty integration remotely (it installs the terminfo but bootstraps via a command, so it skips the MOTD).
- Sources **`~/.bash_secrets`** for secrets (see above).
- No Rust: a previous `curl … | sh` rustup auto-installer block was removed (it could silently reinstall on shell startup). Install toolchains explicitly, never from `.bashrc`.

### Vifm (`user/vifm/vifmrc`)
Wayland-first file manager. The colorscheme is the **GENERATED** `colors/hestia-{dark,light}.vifm` pair (render.py, M7 PR2; the bootstrap symlinks the `theme_variant` one to `colors/hestia.vifm`, so vifmrc's `colorscheme hestia` is variant-blind; on light the common file-type groups drop from the bright to the base ANSI slots for readability). Images → `imv-wayland`, video → `mpv`, PDFs → `zathura`, office docs → `libreoffice`, **Markdown** → **glow** (`:file` offers "Edit in vim"). **Audio** opens in **cmus** (`cmus-remote -f`, play-now) and falls back to `mpv --no-video` when cmus isn't running; `:file` offers queue / mpv alternatives. Preview pane (`w`, on by default) uses `bat` (code, hestia theme) / `mediainfo` (media + image info) / `pdftotext` (PDF) / `glow` (markdown); images are info-only in the pane and open in `imv` (in-pane rendering tried twice — kitty graphics, then viu half-blocks — and dropped both times; the block mosaic wasn't worth the info text); **directories preview as a coloured `tree -C -L 2`** (LS_COLORS — so it matches `ls` and vifm's own file-type colours by construction; `.git` pruned; flat-`ls` fallback if tree is absent). Since 2026-07 (ranger/yazi-inspired, from the file-manager evaluation) the browsing pane uses **miller columns** — `set milleroptions=lsize:1,csize:2,rsize:0` + `set millerview` — a slim parent-context column left of the file list, so with quick view on the layout reads as the ranger triple *parent | current | preview*; `rsize:0` keeps miller's own in-pane preview column off (the quick-view pane already previews everything, dirs included). Clipboard yank (`yd`/`yf`) uses `wl-copy`. Comment syntax is `"` (vim-style).

**Git branch-vs-main helpers** (four commands, base auto-detected main→master, all in `vifmrc`; the pipeline-bearing two share `user/vifm/scripts/git-changed.sh`): `:gd` diffs the **cursor file** vs the branch base in vimdiff; **`:gc`** (key `gc`) opens a **custom-view jump list** of exactly the files this branch changed (Enter opens); **`:gs`** (key `gs`) **highlights** those files *and the folders leading to them* inline in the normal browser via `:select`; **`:gr`** (key `gr`) `cd`s to the **repo root** — handy on its own and the tidy way out of the `:gc` list (or the built-in **`gh`** key / its **`:gh`** alias returns to where you opened it). The `:gd`/`:gc`/`:gs` take an optional base ref like `:gd origin/main`. Several vifm constraints shaped these: a `|` in a `:command`/`:select` is parsed as a command separator (so the pipeline is in the script, not inline); vifm clears the pane selection on directory load (so **auto-highlight via `autocmd DirEnter` doesn't stick** — the `gs` key is the one-press substitute; note `:gs` reuses the selection that file ops like `dd`/`yy` act on); a macro wrapped in single quotes is **not** expanded (so `:gd` uses bare `%c:p`); vifm 0.14 has no `trim()`/`substitute()` (so `:gr` strips `system()`'s trailing newline in-shell with `tr -d`, `|| echo .` making it a no-op outside a repo instead of jumping to `$HOME`); and `gh` is a built-in **key**, not a command, so `:gh` is a `command! gh :normal! gh` alias (typing `:gh` otherwise gave "Invalid command name"). `:gd` resolves the cursor file by **full path → repo-relative** and runs git with `git -C`, so it works from a normal listing *and* the `:gc` custom view (a cwd+basename version showed blank panes there). The catch-all `fileviewer` passes **`%c:p`** (not `%f`): on the `..` parent entry `%f` expands to nothing, shifting a pane dimension into `preview.sh`'s `$1` (the old `cannot open '48'`); `%c:p` is always one absolute path, so `..` previews the parent listing.

The markdown **preview-pane** call is `CLICOLOR_FORCE=1 glow -s ~/.config/glow/hestia.json -w %pw`: when piped (the preview pane is notty) glow drops colour **and** emits malformed empty-param SGR (`ESC[;;1m`) for bold, which vifm's ANSI parser can't read and leaks as literal `1m` in the pane. **`-s` does NOT force colour** (it only carries the style's bold); **`CLICOLOR_FORCE=1` does** — it makes glow emit well-formed SGR (like `bat --color=always`) that vifm renders cleanly. Caveat: forced notty output is **16-colour** (no truecolor), an approximation of the hestia hex; `-s` keeps that mapping on-theme. The full-screen **open** action (`glow -p`) runs in a real TTY, so it gets truecolor and the default style from `glow.yml`.

### imv (`user/imv/config`)
Minimal image viewer config — shrink-to-fit scaling, vim-style `hjkl`/`np` navigation, `HJKL` to pan when zoomed.

**vifm↔imv integrated image + video browsing** (`user/vifm/scripts/imv-browse.sh`, `user/imv/config-vifm`, `user/imv/imv-vifm-return.sh`). In vifm, **opening an image OR video** (Enter/`l`, or vifm's built-in `i` = "handle file" — the image *and* video `filextype` handlers both point here) runs `imv-browse.sh`: it collapses vifm to one pane (`vifm --remote -c only`), `swaymsg split horizontal`, and opens **imv tiled to the right** on the folder's media, started at the cursor file — giving fast full-screen browsing next to vifm (since in-pane graphics never worked here: vifm's preview pane can't render the kitty graphics protocol on this sway/kitty, and previewing per cursor-move lags — so media stays text-info in the pane and *this* is the "see it" path). imv uses a **dedicated config** (`config-vifm`, separate so standalone imv is untouched) that wires it back to vifm over imv's IPC + vifm's client-server (every `--remote` call targets **the launching instance** via `--server-name "$VIFM_SERVER_NAME"` — vifmrc exports it from `v:servername`, and imv inherits it through imv-browse.sh; a bare `--remote` reaches whichever vifm registered the first server name, so with several vifms open the collapse/restore/sync landed in the wrong window — seen live as "the preview pane no longer hides"):
- **Live cursor sync — TWO-WAY (vifm→imv since 2026-07):** imv→vifm is event-driven — each `j/k` is `next/prev 1;exec …/imv-vifm-return.sh`, which `vifm --remote -c "goto '<original>'"`s, so vifm's cursor tracks imv. **`:goto` selects without opening** (a plain `vifm --remote <file>` *opens* the file → would relaunch imv in a loop, and was observed to wedge the server). Two imv gotchas baked in: the `;` has **no leading space** (imv's INI parser strips ` ;…` as an inline comment, silently dropping the chained command), and the logic is in a **script** not inline (imv mangles inline shell quoting in a bind). The reverse — browse in **vifm** and imv follows, so vifm's `/search`+`n`, marks, `gs` etc. stay usable as a media-preview driver — can't be event-driven (vifm's only autocmd is DirEnter), so a detached **watcher** (`imv-browse.sh __watch <dir>`, PID-locked, spawned at launch, exits with imv) polls the launching vifm's cursor via `--remote-expr 'expand("%c:p")'` (~5ms a call, 0.15s cadence) and `imv-msg goto`s imv to it. **Gotcha: `expand()` returns macro values SHELL-ESCAPED** (`my\ pictures`) while handler args / the session map hold raw paths — the watcher sed-unescapes (`s/\\\(.\)/\1/g`) before comparing; unfixed, every comparison failed in any folder with a space, and the dir check closed imv the instant it opened there. Anti-feedback-fight: the return script stamps each imv-originated goto in a **marker file** the watcher consumes-and-skips, and the watcher only acts on a cursor stable for **two consecutive polls** — a fast imv-side `j` run never presents a stable unmarked position, so the watcher can't drag imv backwards. The watcher is also the session's **chaperone**: vifm **leaving the folder** (`expand("%d")` ≠ the session dir) restores the dual-pane preview and **closes imv**; a cursor parked on **anything that isn't this session's media** (unsupported file, a directory, `..`) steers imv onto a trailing **blank "no preview" tile** (plain-ground `blank.jpg`, appended as one extra last imv item so map indexes stay 1:1) — visibly empty rather than frozen on the last image, and no relaunch cost; the return script never syncs vifm onto the blank tile.
- **`q`/`Escape`:** `…/imv-vifm-return.sh quit` — restore vifm's dual-pane preview (`vsplit`+`view!`), sync the cursor, then close imv (`imv-msg $imv_pid quit`, with a `kill` fallback — imv's `exec` swallows the rest of the bind line, so a trailing `;quit` never reaches imv).
- **`h` (up/back, vim-style):** `…/imv-vifm-return.sh up` — close imv and take vifm **up one directory**, cursor on the folder you were browsing. Uses `:goto <dirname-of-image>` (absolute), **not** `cd ..`: the user can change vifm's cwd while imv is open, so the parent is computed from the image's path, not vifm's current dir. (j/k/l/n/N/p still cycle images; prev is on `k`.)
- **Idempotent / re-press, and OTHER imv instances are sacred:** opening another item while **this session's** imv is up doesn't stack a second pane — it `imv-msg $pid goto <index>`s it and focuses **by PID** (`[pid=N]`, not `[app_id="imv"]`); if the item **isn't in the open session** (vifm moved to another directory, or the file is new), the old imv is retired and a fresh session launches on the current dir. The session's imv is identified by a **pidfile, never pgrep**: the launcher `exec`s imv so its own `$$` *becomes* imv's PID (recorded per server name, `/proc/<pid>/comm`-validated against PID reuse) — a bare `pgrep imv-wayland` matches ANY imv, and the teardown used to close a standalone imv / another vifm window's session (caught live). ALL session state is **namespaced by `$VIFM_SERVER_NAME`** (map, locks, marker, pidfile; the thumb cache + backfill worker stay global), so browse sessions in different vifm windows coexist. (This is the whole reason the handler is `imv-browse.sh` rather than a bare `imv` — a plain open spawned a confusing second pane.) imv 4.5 can't sort, so imv-browse.sh launches imv on an explicit list in **vifm's exact view order**: the python pass replicates vifm 0.14's `+name`+`sortnumbers` comparator byte-for-byte (sort.c/utf8.c — NFKD compat-decomposition of the name, skip-leading-zeros, then **glibc `strverscmp` called via ctypes**, the same libc function vifm links, not a reimplementation), verified identical against a live vifm on an adversarial name set. (`sort -V` was the first cut, but coreutils filevercmp ≠ strverscmp — its extension rule, no unicode normalization — and every divergence made the synced cursor **jump**; vifm's order isn't queryable over `--remote` — `expand("%a")` is empty — so we reproduce it.)
- **Videos are faked as images.** imv can't play video, so imv-browse.sh feeds it a **cached ▶ poster thumbnail** for each video (`ffmpegthumbnailer -s 1280` → `magick` composites a play button scaled to the frame; ffmpeg fallback; cached by realpath+mtime). **Priority order (2026-07, after a 2000-video dir wedged the flow):** only the CURSOR file's poster is generated synchronously; other uncached videos show a generic ▶ placeholder and imv launches immediately; a single `nice`/`ionice` background worker (`__backfill` self-invocation, PID-locked) fills the cache sequentially for the next visit. The display list is mapped in **one python pass** (`build_lists`), not a per-file shell loop — the loop's ~6 forks per file were an *independent* ~7s hang at 2000 files even with zero thumbnail work — and that same pass pre-filters the backfill queue to just the missing posters, so a fully-cached dir spawns no worker at all. A PID-checked **launch lock** makes an Enter-storm harmless (the pgrep-imv check can't see the pre-imv window — repeated Enters used to stack launchers). imv only ever sees images. A **session map** (`$XDG_RUNTIME_DIR/imv-vifm-session.<servername>.list`, one `display<US>original` pair per line) resolves imv's current item back to the real file **by display path (`$imv_current_file`), NOT by index**: imv silently *drops* entries it can't load (one corrupt image / unsupported format), shifting every later index — index-resolution made each poster after the bad file "belong" to the wrong video (caught live 2026-07). For the path to be a unique key, each uncached video gets its **own placeholder *hardlink*** (`<thumbkey>.pend.jpg`, pruned at next launch; symlinks don't work — imv canonicalizes them back to `placeholder.jpg`). The cursor sync `:goto`s the **video** (not its thumbnail), and…
- **Enter on a video → mpv** (`…/imv-vifm-return.sh play`): opens mpv **under the vifm+imv combo** (`swaymsg focus parent, split vertical` wraps `[vifm|imv]` and tiles mpv below). **Single-instance, or it fork-bombs:** as focus returns to imv when mpv closes, a spurious Enter (sometimes a train) re-fires the bind. A **PID lock** (`imv-vifm-mpv.lock`) gates it — while a launch/mpv is alive, re-fires just focus the existing mpv; a **debounced cooldown** (the detached wrapper holds the lock until ~2s idle after mpv exits, each re-fire pushing the deadline out) absorbs that train so nothing relaunches; and the lock is **PID-checked** (`kill -0`) so a force-killed mpv leaves no stale lock that dead-ends Enter. (Enter on an image is a no-op — you're already viewing it full-screen. Cost: a ~2s beat after closing mpv before the next video opens.)

True single-window grouping (like the cmus+cava kitty session) isn't possible — imv is a separate Wayland app, not a kitty pane — so the idempotent guard is what keeps the layout from duplicating. Deps: `imv`/`imv-msg`, `swaymsg` (sway), `mpv`, `vifm` server (on by default; `vifm --remote` no-ops if absent), and `ffmpegthumbnailer` + `imagemagick` for the video thumbnails (both in the `terminal` apt group).

### Git (`user/git/`)
Default branch `main`, editor `vim`, commits GPG-signed. The block is Jinja-gated on the toggle — no agent user → no claude-owned tree → no entry rendered. No HTTPS credential helper is configured — auth is over SSH (`git@github.com`), so an HTTPS remote would simply prompt; add `helper = cache` for an in-memory, no-plaintext fallback if ever needed (avoid `helper = store`, which writes plaintext to `~/.git-credentials`). - **Launcher / NVIDIA workarounds (`system/sway-session/start-sway`):** sway isn't started by the config above — greetd runs **`/usr/local/bin/start-sway`** (`tuigreet … --cmd start-sway`), which sets the Wayland session env then `exec sway`. The NVIDIA-specific env (`GBM_BACKEND=nvidia-drm`, `__GLX_VENDOR_LIBRARY_NAME=nvidia`, `WLR_NO_HARDWARE_CURSORS=1`, `NVIDIA_DRIVER_CAPABILITIES=all`) + `--unsupported-gpu` are applied **only when an NVIDIA GPU is live** (guarded on `/dev/nvidia0` / the `nvidia` entry in `/proc/modules`) — on AMD/Intel they'd break rendering, so they're skipped. Both `start-sway` and the **greetd `config.toml`** that calls it are tracked (`system/sway-session/`) + deployed by the **`sway_session`** role (greetd config → `/etc/greetd/`, applied on next login — the role doesn't restart greetd). Only installing greetd/tuigreet themselves is a prereq (`system/sway-session/README.md`).

- **Idle inhibition (the "caffeine" replacement):** the auto-lock chain is `swayidle` (**pre-lock dim warning at 270s** → lock at 300s → DPMS-off at 600s) → `swaylock` (now reads `~/.config/swaylock/config`, hestia-themed — see the **Swaylock** section; the invocations carry no `-c`, so the config's `color` is the lock background). The **270s warning** dims *every* output to 40% via a **backgrounded `gammastep -O 6500 -b 0.4 &`** software-gamma drop (killed by `pkill -x gammastep` on the timeout's `resume`) **and** raises a persistent `notify-send -u critical -t 30000` notice (matches mako's `[urgency=high]` red style, stays the full 30s — previously it used the 5s default-timeout, so the warning vanished 25s before the lock). **gammastep must be backgrounded:** on Wayland its `-O` "one-shot" doesn't actually exit (wlroots resets the gamma ramp the instant the setting client disconnects, so gammastep stays alive to *hold* the dim) — and `swayidle -w` waits for each timeout command to finish, so a foreground gammastep would wedge swayidle forever (no resume fired, no lock). Backgrounding lets `notify-send` return; `resume` then `pkill`s gammastep (the disconnect restores brightness) **and dismisses the warning notice** — the `-t 30000` would otherwise leave it on screen for the full 30s after input already cancelled the lock, so `notify-send -p` stashes the notification id in `$XDG_RUNTIME_DIR/lock-warn.id` and `resume` runs `makoctl dismiss -n <id>` to clear that one (not `-a`, which would drop unrelated notifications on every wake). The dim is the real cue: this is a **desktop with external DisplayPort monitors and no `/sys/class/backlight`**, so `brightnessctl` can't dim the panels (the `XF86MonBrightness` keybinds are no-ops on this hardware) — software gamma dimming hits both 32" screens at once regardless of which has focus. Any input before 300s fires the resume (full brightness back, no lock). gammastep is in the apt manifest's `wayland` group. To stop it locking during fullscreen video, sway's native `inhibit_idle fullscreen` is set per media app — `for_window [app_id="firefox"|"chromium"|"mpv"|"^FFPWA-"] inhibit_idle fullscreen` — so those windows raise a Wayland idle inhibitor *while fullscreen* (swayidle honours it; the timers pause, then resume on un-fullscreen). Deliberately **scoped to media apps**, not `[app_id=".*"]`: a blanket rule would let a forgotten fullscreen terminal skip the auto-lock (the security boundary). `app_id` is unanchored regex, so `firefox` also matches `firefox-esr`, and **`^FFPWA-` matches every `firefoxpwa` web app** (Netflix, YouTube, …) — each PWA gets its own `FFPWA-<ULID>` app_id (not `firefox`), so the prefix match is what actually catches the common "fullscreen Netflix in a PWA" case; add a browser = one more line. (mpv also self-inhibits via `--stop-screensaver` by default; the sway rule is the belt-and-suspenders that doesn't depend on the app speaking the `idle-inhibit` protocol.)

Apply most config changes with `swaymsg reload`.

### Swaynag (dialogs) (`user/swaynag/config-{dark,light}`)

### Screen sharing / xdg-desktop-portal (`user/xdg-desktop-portal/sway-portals.conf`)
Wayland screen sharing (Teams/Edge, OBS, etc.) goes through **PipeWire** + **xdg-desktop-portal**. On Sway the screencast backend is **`xdg-desktop-portal-wlr`** (wlroots); other portal interfaces (file picker, settings) fall to **`xdg-desktop-portal-gtk`**.

- **`sway-portals.conf`** routes interfaces per backend (matched to this session via `XDG_CURRENT_DESKTOP=sway`): `ScreenCast`/`Screenshot` → `wlr`, everything else `default=gtk`. Without explicit routing the portal can misroute screencast to gtk (which can't capture on wlroots) and silently fail. If gtk is ever uninstalled, the `default` must change (wlr only implements ScreenCast/Screenshot — no file picker).
- **Session env:** the portal must inherit `WAYLAND_DISPLAY`/`XDG_CURRENT_DESKTOP` via the Sway `dbus-update-activation-environment` exec line (see Sway section). The portal is **D-Bus-activated on demand** (first screencast request), so the env must be exported *before* it first activates — a fresh login handles this.
- **Monitor-only capture:** the wlr backend reports `AvailableSourceTypes=1` (MONITOR only) — **per-window sharing is not supported** on Sway; share a whole output. Workaround: isolate the app on the second monitor and share that output. May change once the `wlr` portal adopts `ext-image-copy-capture` toplevel sources.
- **Dark mode (color-scheme):** the gtk backend exposes `org.freedesktop.appearance color-scheme` via the **Settings** portal, which Firefox and other portal-aware apps read. Force dark with `gsettings set org.gnome.desktop.interface color-scheme prefer-dark` (needs `dconf-cli` + `libglib2.0-bin`); without it the portal reports `0` (no preference) and apps default to **light**. This surfaced right after the portal was first installed — Firefox flipped to a light theme until the setting was applied. **Now set at login** by the sway theme fragment's `exec` (per `theme_variant` — `prefer-light` on the light desktop), so it's reproducible — see the **GTK / Adwaita** section.

**External dependencies** (install on a fresh system):
- `pipewire`, `pipewire-pulse`, `wireplumber` (usually present; the audio stack).
- `xdg-desktop-portal` + `xdg-desktop-portal-wlr` (screencast) + `xdg-desktop-portal-gtk` (file picker/other portals).
- `dbus-bin` provides `dbus-update-activation-environment` (used by the Sway env-export exec line).

### Waybar (`user/waybar/config`, `user/waybar/style.css`, `user/waybar/scripts/`)
Top bar; files symlinked into `~/.config/waybar/`. Reload with `killall -SIGUSR2 waybar` (or `killall waybar; waybar &`). Colours are `@define-color` tokens in the **GENERATED** `theme-{dark,light}.css` pair (render.py), `@import`ed by `style.css` as `theme.css` (the bootstrap symlinks the `theme_variant` one) — edit `palette.yml` + re-render, not the fragment; `style.css` itself references only `@token` names.

**Single launcher + boot race:** waybar is started by sway's `exec_always` line running `start-waybar.sh` and must be the *sole* launcher. Two boot problems, two fixes: (1) a globally-enabled `waybar.service` user unit double-started it — disabled with `sudo systemctl --global disable waybar.service`; (2) even solo, waybar launched before the compositor could map its layer-shell surface, so the bar didn't render until a `$mod+Shift+c` reload. `user/sway/start-waybar.sh` fixes (2) — it kills any instance, settles, and waits for an active output before `exec waybar`. (The session is greetd-launched, not systemd-managed, so a `graphical-session.target`-ordered user service isn't an option without extra plumbing.)

- **Font:** the **bar** uses `BigBlueTermPlus Nerd Font Mono` at 16px (`style.css`, the `*` rule). Bitmap-derived — keep its sizes at integer pixel-grid values (16/24/32) for crispness. **Not** the `437` variant: that one is the literal DOS code-page-437 build, which has no Greek *alphabet* (only ~12 Greek/math *symbol* glyphs — α π Σ Ω δ ε σ τ …), so a non-Latin window title (`sway/window`) rendered its `ε` as a euro-like lunate epsilon and fell back mid-word for every letter CP437 lacks. `Plus` is the same bitmap typeface with full Greek (and Cyrillic etc.) plus the same Nerd Font icon glyphs, so the look is unchanged but titles render cleanly. **Tooltips** are restyled via `tooltip`/`tooltip label` rules: `Lilex Nerd Font Mono` 17px (scalable, cleaner than the bitmap bar font, also full Greek coverage), on the unified `#0a0a0a` bg with a 1px `#d7005f` border + padding (GTK tooltips are otherwise translucent/borderless).
- **Icon alignment:** module icons are wrapped in inline Pango markup `<span size='xx-large' rise='-3072'>…</span>`. The `rise` is a hand-tuned vertical offset (Pango units, 1/1024 pt); re-tune if the font changes.
- **PUA glyph caveat:** editing tools can silently strip Private-Use-Area (Nerd Font) glyphs, leaving an empty `<span></span>`. If an icon renders as blank space, insert the codepoint via a small Python write and verify, rather than re-typing it.

Right-side modules share styling via a grouped CSS selector — add new ones to that group.
- **Fixed-width numeric values** keep the bar from resizing as values change width (e.g. `0%`→`100%`): percentages are right-padded to **2 digits** — custom modules in the script (`printf '%2d%%'` in `cpu.sh`/`gpu.sh`), built-in modules via fmt spec in the format (`{signalStrength:>2}`). (2 not 3: it reserves up to `99%`; the rare `100%` adds one char — an acceptable jump.) Leading spaces render fine in the bar label.
- **Click handlers** open floating TUIs as toggles so a second click closes the popup. Two toggle styles: for a known process name, `pgrep -u $USER -x NAME && pkill -u $USER -x NAME || kitty --class floatterm -e NAME` (network/bluetooth/audio) — **always `-u $USER`-scoped**: bare pgrep matches other users' processes too, and the `claude` agent runs these same apps headlessly on this box (caught live 2026-07: a leftover claude `imv-wayland` made vifm's image-open silently no-op). These sway-tree popups launch kitty with `-o confirm_os_window_close=0` so closing them (the WM kills the window while a program runs) doesn't pop kitty's confirm dialog — the `pgrep -x` toggles don't need it because they `pkill` the process first, so kitty exits cleanly.

### GTK / Adwaita (`user/gtk/`, `bootstrap/roles/gtk_theme/`)
GTK theming for the unified look (`docs/theming.md`). The hard-won outcome after a deep investigation: **GTK3 apps get dark + the exact `#d7005f` accent; GTK4/libadwaita apps get dark only** (their accent is locked on Debian's libadwaita — see below). Pieces:
- **The hestia GTK3 theme = recoloured adw-gtk3.** libadwaita-matching GTK3 theme **`adw-gtk3`** (not in Debian apt) is downloaded pinned (`v6.5`, sha256-verified) by the **`gtk_theme` role**, which seds its accent `@define-color accent_bg_color @blue_3` → **`#d7005f`** and installs it as `~/.local/share/themes/hestia` + `hestia-dark`. No compiler: adw-gtk3 references `@accent_bg_color` at **runtime** (156×), so one `@define-color` edit recolours every accent surface (focus rings, selection, tabs, shaded hovers). Idempotent via a `(version, accent)` marker. This covers GTK3 apps — **Remmina, GIMP, Firefox's file chooser**.
- **Selected by `GTK_THEME`, with gsettings for the portal.** The decisive selector for in-process GTK3 apps (Remmina, GIMP) is **`export GTK_THEME=hestia{-dark}` in `start-sway`** (templated per `theme_variant` since M7) — `GTK_THEME` is the highest-priority theme source and **overrides both gsettings `gtk-theme` and `~/.config/gtk-3.0/settings.ini`** (in a non-GNOME sway session those two turn out to be *ignored* for the theme name — a hard-learned gotcha: the launcher's old `GTK_THEME=Adwaita:dark` was silently forcing stock Adwaita and masking every gsettings/settings.ini/gtk.css change we made). Separately, the sway **theme fragment** (`include`d at the top of the config) `exec`s the variant's gsettings — `color-scheme prefer-dark|light` (portal-aware/Firefox/libadwaita), `gtk-theme hestia-dark|hestia` (the **portal file chooser** reads gsettings, not `GTK_THEME`/settings.ini) **and `cursor-theme`** — plus a `seat * xcursor_theme` line for sway's own cursor. The **mouse cursor is KDE Breeze** (`breeze-cursor-theme`, apt): white `Breeze_Light` on the dark desktop, standard `breeze_cursors` on light — selected in three places that must stay in sync (the fragment's seat+gsettings, `settings.ini.j2`, and start-sway's `XCURSOR_THEME`/`XCURSOR_SIZE` export for XWayland/Qt/SDL). `gtk-3.0/settings.ini` (now **rendered** from a `.j2`, theme name per variant) + the `gtk-3.0/gtk.css` overlay are kept belt-and-suspenders.
- **GTK4 / libadwaita accent — a dead end on trixie, but it's a *version* wait, not a theming dead end.** Proven (three ways, cleanly isolated) that **libadwaita 1.7 locks the accent against user CSS** — neither `@define-color accent_bg_color` nor the `:root { --accent-bg-color }` form recolours gnome-calculator (the file *loads* — a green-window control showed — but the accent doesn't budge), and libadwaita ignores external themes entirely. **The lever is the libadwaita version, not the theme.** libadwaita **≥1.8** moved its colours to GTK CSS custom-properties and accepts an **arbitrary hex** accent (the [styles doc](https://gnome.pages.gitlab.gnome.org/libadwaita/doc/main/styles-and-appearance.html)): `:root { --accent-bg-color: #d7005f; --accent-color: oklab(from var(--accent-bg-color) var(--standalone-color-oklab)); }`. That's exactly what `gtk-4.0/gtk.css` stages — **inert on 1.7, auto-active at ≥1.8** — so on the move there, gnome-calculator/nautilus get exact `#d7005f` **from our own CSS, no Ubuntu patch and no Yaru needed**. Versions (Debian tracker, 2026-06): oldstable **1.2.2** (pre-accent — so *older* Debian is strictly worse), trixie **1.7.6** (frozen; stable never major-bumps), testing/forky + sid **1.9.1**. So the wall falls on the upgrade to **Debian 14**, not before — *don't* franken-mix sid's libadwaita onto trixie. **Refinement for when it activates:** make the staged `:root` set **`--accent-color` standalone too**, not just `--accent-bg-color` (per the doc). The portal `accent-color` stays moot (nine-colour named enum; `xdg-desktop-portal-gtk 1.15.3` doesn't export it). Until Debian 14, GTK4/libadwaita apps are **dark with the default blue accent**.
- **Yaru was evaluated (2026-06) and scoped to *icons only*.** Yaru is a maintained GTK+icon suite with a custom-accent SCSS/meson build, but: for **widgets** it's *heavier* than our one-`sed` adw-gtk3 recolour and gives up the exact `#d7005f`; for the **GTK4 accent** it doesn't help on Debian — Yaru's libadwaita accent rides **Ubuntu's downstream libadwaita patches** (`style-manager-Support-Yaru-accent-colors.patch`, …) that Debian doesn't carry, so it hits the same 1.7 wall (and ≥1.8 makes it unnecessary anyway — see above). Where Yaru *does* win is **icons** (Suru): adw-gtk3 themes none, and icons are **immune to the libadwaita lock** (just files). So the **`yaru_icons` role** (opt-in, `enable_yaru_icons`, default off) installs a custom **`Yaru-hestia`** theme into `~/.local/share/icons` — but as a **prebuilt artifact, not build-on-deploy**: there's **no sed-shortcut** (Yaru ships accented folders as rasterised PNGs, so the accent must be *rendered* via `meson`/`ninja`/`sassc`/inkscape+xvfb — too heavy/fragile per machine). Instead the **`claude` agent builds it once on demand** and publishes `Yaru-hestia-icons.tar.gz` (base `Yaru` + accent overlay, ~27 MB) as a hestia **release** (`yaru-hestia-v2`); the role just **downloads + sha256-verifies + extracts** it (like `localbin`). Full build/maintenance recipe: `bootstrap/roles/yaru_icons/README.md`. Selected via `gtk-icon-theme-name=Yaru-hestia` in both `settings.ini` + a **guarded** sway `gsettings` exec (set only if the theme exists, so a no-role spin falls back cleanly). **Accent gotcha:** Yaru paints the folder front through a ~15% white overlay that lightens the accent, so it must be **pre-compensated to `#d00043`** — the rendered front then lands on true `#d7005f` (≈`#d7265f`; R/B exact, green floors at ~38), verified by sampling a rendered `places/folder.png`. (v2 is rebuilt for hestia red; the former Debian-red v1 used `#c60039`.) We ship the **light** flavour and use it even on the dark desktop (Yaru's *dark* variant re-lightens folders to pink; folders are bg-independent, symbolics recolour via GTK). Keep adw-gtk3 for widgets; KDE/Qt stays a separate **Kvantum** track.
- **Firefox & its `firefoxpwa` PWAs follow hestia-dark.** They're GTK3, so `GTK_THEME=hestia-dark` themes their **chrome, menus, and file chooser** to the red accent — no `userChrome` needed (the earlier "Firefox is out of reach" worry was just the `GTK_THEME=Adwaita:dark` bug masking it; each PWA is a `FFPWA-*` GTK3 window and picks it up too). Dark web *content* comes from `color-scheme`; the only conceivable holdout is in-page form controls drawn by Firefox's own engine, which `userContent.css` could address if ever wanted.
- **Deferred (gradual):** base/surface colours toward the exact hestia palette (`#0a0a0a`/`#111111`/…) in the theme; **Kvantum/Breeze** (the Qt/KDE companion); and the libadwaita accent (auto-resolves at Debian 14 / libadwaita ≥1.8). The **custom `#d7005f` Yaru *icon* flavor** is **done** — opt-in `yaru_icons` role, rebuilt as `yaru-hestia-v2` (folder render verified at `#d7265f`). Status table in `docs/theming.md`.

### Thunderbird (`user/thunderbird/`)
Email client, hestia-accented (verified live on 140.12 ESR, 2026-07). The GTK theme only reaches its **native GTK widgets** (context menus, file chooser); the main window is Gecko-rendered from **three separate variable chains**, all blue by default — traced by extracting the stylesheets from `/usr/lib/thunderbird/omni.ja`:
1. **`--selected-item-color: var(--sidebar-highlight-background-color, AccentColor)`** (messenger's `variables.css`) — selection, primary buttons, focus rings, tabline, spaces toolbar all derive from it. `AccentColor` itself **ignores the GTK3 theme**: Gecko ≥~126 takes it from libadwaita's palette + the GNOME accent enum (`widget.gtk.libadwaita-colors.enabled`, default true) — the Gecko flavour of the libadwaita-1.7 accent wall in the GTK section.
2. **Hardcoded blues that derive from nothing:** `--primary` (`#0a84ff`), `--color-primary-{soft,default,hover,pressed}` (account hub), `--color-text-highlight`, `--button-link-text-color`.
3. **The toolkit design-system token `--color-accent-primary`** (`tokens-brand.css`) — the Settings tab (`about:preferences`) never loads messenger's `variables.css`; its checkboxes/radios paint from the CSS **`accent-color`** property, set from that token. A `userChrome` override can't reach it — about: pages are *content* documents (`userContent.css`).

The fix is three tracked files in `user/thunderbird/`: **GENERATED** `userChrome.css` + `userContent.css` (render.py — **single files carrying both variants via CSS `light-dark()`**, since TB follows the portal color-scheme; the one render.py target that isn't a dark/light pair) and a plain `user.js` pinning `toolkit.legacyUserProfileCustomizations.stylesheets=true` (loads the chrome CSS) + `widget.gtk.libadwaita-colors.enabled=false` (AccentColor back to the GTK3 theme for content selection/scrollbars). **Shadow-DOM gotcha:** `userChrome.css` targets **`*`, not `:root`** — the unified-toolbar tabs are custom elements whose shadow-root stylesheets re-declare the variables on `:host` (a declared value beats one inherited from `:root`); a user-origin `!important` declared **on the host element itself** — which `*` matches — wins across the shadow boundary. **Deployment:** the profile dir name is randomized (`~/.thunderbird/<hash>.<name>`), so these can't ride the fixed-dest `dotfile_links` manifest (they're absent from the Active-symlinks table by design) — the dotfiles role's `thunderbird.yml` **globs** for any profile holding a `prefs.js` and symlinks all three files into each (no-op when TB has never run; fresh install = start TB once, re-run `--tags dotfiles`, restart TB). The `thunderbird` **package** is the opt-in `email` apt group (`enable_thunderbird`, default false, setup.sh asks) — the theming follows TB's *presence*, not the toggle.
