# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

Personal dotfiles for a Debian-based Linux system running the **Sway (Wayland)** desktop. No build system, no tests, no linting — changes take effect by editing the files directly (the active ones are symlinked into `~` / `~/.config`). Reproducing the machine from scratch is driven by an **Ansible bootstrap** (`bootstrap/`, now five roles — see *Bootstrap & system reproduction*); the ordered fresh-install runbook is `docs/install-runbook.md`.

The repo is growing from *one person's dotfiles* toward a **reproducible Debian spin** with multiple principals and `/etc` system configs; the target layered structure is `docs/repo-structure-design.md` (phased, not yet migrated).

The configs are converging on one unified theme: near-black background `#0a0a0a`, accent **official Debian red `#ce0056`** (PANTONE Strong Red C), and a saturated 16-colour palette derived from the **`wildcharm`** vim colorscheme. The palette is now defined once in **`themes/wildcharm/palette.yml`** (the single source of truth — semantic roles + the 16 ANSI colours), and the **per-app theming process** (+ a themed/not-yet status table) is **`docs/theming.md`**. Keep new theming consistent with this — and when theming an app, follow that process.

## Running as the `claude` agent user

Claude Code runs here as a dedicated, unprivileged **`claude`** Linux user (own `0700` home `/home/claude`, member of the **`devshare`** group, **not** in sudo) — a kernel-enforced trust boundary, not a behavioural one. Full design: `docs/claude-user-design.md` (implemented 2026-06-20); day-to-day workflow: `docs/working-with-claude.md`. What matters when working in this repo:

- **This checkout (`/srv/devshare/hestia`) is claude's own clone** in the shared `/srv/devshare` tree — claude's *workspace*, **not** the live system. The dotfiles are symlinked from **dimitrios's** clone at **`~/Development/hestia`**, which is the deployment source. Edits made here reach the running system only when **dimitrios pulls** after reviewing a pushed branch — so an edit here is **not** live. Never treat `/srv/devshare/hestia` as the deployment source. (No `/home/dimitrios/...` literals remain in any deployed config — every host-specific path resolves from `target_home`/`repo_root`/`$HOME`/host_vars at deploy time. Path generalisation is **complete**, `docs/repo-structure-design.md` §5.)
- **Collaboration is git-mediated, two principals.** claude commits and signs as **itself**, pushes a topic branch, and opens a **pull request** as the GitHub bot **`dimitrios-claude`** (`gh pr create`; commits show **Verified**); dimitrios reviews, **merge-commits** (preserves the signed commits; via `gh pr merge --admin`), and pulls to deploy. claude **never** self-merges — and *cannot*: a `main` ruleset (*Restrict updates*, admin-only bypass) blocks the agent from pushing to or merging `main` server-side, so its only route is opening a PR. The concrete loop (and the `gh`/PAT auth) is in `docs/working-with-claude.md`.
- **claude's identity:** own SSH key `~/.ssh/id_claude`, own **passwordless** GPG key `4AA9DD310356AD0E`, git author `Claude (dimitrios's agent) <claude@charalampidis.pro>`.
- **Signing is frictionless for claude** — the passwordless key needs **no pinentry, no agent cache, no `gpg-unlock`, and not `gpg-wrapper.sh`**. All that machinery in *Secrets & commit signing* below is **dimitrios's**; it does not apply when running as claude.
- **Isolation:** claude **cannot** read dimitrios's home, `~/.ssh`, `~/.gnupg`, or `~/.bash_secrets` (different UID; `0750`/`0700`/`0600`). Confirm any time: `ls ~dimitrios` → Permission denied.
- **`user/bin/claude-access`** (`grant|revoke|list`, symlinked to `~/.local/bin/`) is the escape hatch to give claude ACL access to a path **outside** `/srv/devshare` (e.g. a project in the human's home), with a tracked registry. Prefer the clone-and-sync model; reach for it only when in-place editing is truly needed.
- **claude can install its own user-level tooling.** Because `claude` is a separate user with its own `/home/claude`, it can freely install whatever a task needs **into its own home** — npm/pip `--user`, cargo, a downloaded binary into `~/.local/bin`, etc. — without `sudo` and without touching dimitrios's environment. (System-wide `apt` installs still need the human; those stay dimitrios's job.)

## File layout and naming convention

Top level: **`user/`** (the human's home-deployed configs — one subdir per tool), **`system/`** (root-owned `/etc` configs), **`bootstrap/`** (the Ansible installer), **`themes/`** (the theme single-source-of-truth — `wildcharm/palette.yml`; see *Theming* / `docs/theming.md`), **`docs/`**. The `user/` grouping is a flat namespace — *not* `users/<name>/`: per-user values are parameterised (host_vars/templating, §6), so a username sub-level would be empty ceremony (see `docs/repo-structure-design.md` §7).

Active `user/` configs are either:
- **Dot-prefixed** in the repo, symlinked into `$HOME` — `user/vim/.vimrc` → `~/.vimrc`, `user/bash/.bashrc` → `~/.bashrc`, `user/git/.gitignore_global` → `~/.gitignore_global`.
- **Plain files under the app dir**, symlinked into `~/.config/<app>/` — sway, waybar, mako, wofi, cava, cmus, kitty, imv, vifm, glow (`wildcharm.json`), nvim, xdg-desktop-portal.
- **Rendered (not symlinked)** — a small **path-generalised / identity-bearing** set rendered from `.j2` by the bootstrap (`user/git/.gitconfig`, `user/glow/glow.yml`, `user/cmus/rc`, and the two credential scripts `credential-unlock.sh` + `keyring-ssh-askpass.sh`); see *Rendered (templated) configs* below.
- **`user/bin/`** — small helper scripts symlinked into `~/.local/bin/` (currently `claude-access`; see *Running as the `claude` agent user*).
- **`system/`** — layer-(a) **system configs** deployed by **copy**/template to root-owned locations (**not** symlinked), driven by Ansible roles: `system/samba/smb.conf` → `/etc/` (`samba` role); `system/sway-session/{start-sway → /usr/local/bin/, greetd-config.toml → /etc/greetd/config.toml}` (`sway_session` role). Each `system/` subdir keeps a `README.md` runbook.
- **`bootstrap/`**, **`docs/`** — the Ansible installer + the design docs/runbooks (cross-cutting, not a per-tool config).

Two `/srv` trees live outside the repo but are part of the setup: **`/srv/devshare`** (the human↔`claude` shared dev tree — group `devshare`, setgid + default ACLs; this clone lives here) and **`/srv/smbshare`** (the Samba share — group `smbshare`; `docs/file-sharing-design.md`).

## Secrets & commit signing

> This section describes **dimitrios's** interactive setup (the passphrase-protected key + the pinentry-wedge machinery built around it). When running as the **`claude`** user, signing is **passwordless** and none of the wrapper / cache / `gpg-unlock` flow applies — see *Running as the `claude` agent user*.

- **Secrets** (API tokens, etc.) live in **`~/.bash_secrets`** — untracked, mode 600, sourced at the end of `user/bash/.bashrc`. Never commit secret values; a bootstrap script must not carry this file.
- **Commits are GPG-signed** (the signing key id is an **installer input** — `git_signingkey` in host_vars, rendered into both `user/git/.gitconfig.j2` and `credential-unlock.sh`; empty → `gpgsign = false`). Signing is routed through **`user/git/gpg-wrapper.sh`** (set as `gpg.program` in `user/git/.gitconfig.j2`, templated from `{{ repo_root }}` so it follows the clone — no manual update on move). In a normal terminal it execs `gpg` untouched, using **`pinentry-tty`** (configured in **`user/gnupg/gpg-agent.conf`**, symlinked to `~/.gnupg/`) with a **~session-length cache TTL** (`34560000`) — because login auto-unlock (below) warms the agent at login, so the security boundary is the unlocked session + screen lock, not the TTL; GPG now matches ssh-agent's persist-until-logout.
- **Why the wrapper:** inside Claude Code (`$CLAUDECODE` set) gpg-agent would launch pinentry on `GPG_TTY` — which is Claude's own terminal (`/dev/pts/0`), **not** the calling process's stdin — and **seize it** (you can't type the passphrase or control the prompt; `pinentry-tty` does *not* avoid this, the tty takeover is the problem). So under `$CLAUDECODE` the wrapper adds `--pinentry-mode error`: a **warm** cache signs silently, a **cold** cache fails fast (`gpg: signing failed: No pinentry`, exit 2) instead of wedging. When a harness commit fails this way, run **`gpg-unlock`** (a `user/bash/.bashrc` function — signs a throwaway message so gpg-agent prompts once via pinentry-tty; reads the key from git config, so no hash to remember; refuses to run under `$CLAUDECODE`) in a normal terminal, then the harness can sign again within the cache window.
- **Bootstrap / pinentry default:** the tracked `gpg-agent.conf` forces `pinentry-tty` per-agent, but the *system* `pinentry` alternative defaults to **`pinentry-curses`** (leaks mouse-tracking escape codes). Pin it belt-and-suspenders with `sudo update-alternatives --set pinentry /usr/bin/pinentry-tty`. Avoid `pinentry-curses` and `pinentry-gnome3` (the latter falls back to curses outside GNOME).
- **Login auto-unlock (SSH + GPG):** `pam_gnome_keyring` unlocks the gnome-keyring login keyring at login; the Sway-`exec` hook **`user/gnupg/credential-unlock.sh`** then loads the SSH key (via an SSH_ASKPASS helper reading the keyring) and warms gpg-agent (loopback sign), so **no per-boot `ssh-add`/`gpg-unlock`** — including for the harness. Passphrases live in the keyring (`secret-tool store … autounlock ssh/gpg …`). Full story incl. the gnome-keyring launcher-untangle needed to make PAM the sole unlocker: `docs/credential-autounlock-design.md` §10. The wrapper + `gpg-unlock` remain as fail-clean fallbacks for a cold cache.

## Bootstrap & system reproduction (`bootstrap/`)

A fresh machine is reproduced by an **Ansible** playbook (engine decision: `docs/repo-structure-design.md`). The one-shot entry point is **`bootstrap/setup.sh`** — installs Ansible, interactively gathers host answers (the §6 feature toggles + `samba_lan_subnet` + `cmus_music_dir`, with auto-detected defaults) into the untracked `host_vars/localhost.yml`, then runs the playbook (re-runnable; own flags `--no-backup`/`--yes`/`-h`/`--help`, everything else passes through — and `--check` is a true dry-run: answers go to a temp file, real `host_vars` untouched, ansible only previews). It **captures the sudo password up front with its own retry loop** (validating each attempt via `sudo -S -v`, 3 tries) and hands it to ansible via `--become-password-file` pointing at a **tmpfs file** (`$XDG_RUNTIME_DIR`/`/dev/shm` — RAM-backed, `0600`, removed right after the run), instead of ansible's single-shot `--ask-become-pass` where one typo aborts the whole play. (Two things that *don't* work, learned the hard way: a cached `sudo -v` timestamp — sudo's `tty_tickets` keys it to the shell's tty but ansible's become runs on a different tty; and process-substitution `<(…)` for the password file — ansible re-opens the path by name and `/dev/fd/N` resolves to an unopenable `pipe:[inode]`.) The NOPASSWD probe runs `sudo -k` first so a warm cache can't masquerade as passwordless. **`--yes`/`-y`** skips the questionnaire and reuses the saved `host_vars` as-is (resume a failed run). **It is destructive on a fresh `$HOME`** (the `dotfiles` role force-replaces existing dotfiles) — so a *first run* (no host_vars yet) shows a ⚠️ replace notice (in dry-run too — it's a simulation) and on a real apply requires typing `yes`, and **on the first deploy only** (marker-tracked at `~/.local/state/hestia/.dotfiles-backed-up`) the role copies any pre-existing real config to `<file>.bak` next to it — one `.bak` per file, ever, no accumulation. Disable with `--no-backup` (= `-e dotfiles_backup=false`); then replaced configs aren't saved. The ordered start-to-finish narrative — which role when, with the interactive/external manual steps interleaved — is **`docs/install-runbook.md`**; the per-role reference is `bootstrap/README.md`.

- **Single source of truth: `bootstrap/group_vars/all.yml`** — the manifest listing every dotfile symlink (`dotfile_links`) and the apt package set (`apt_packages`). The "Active symlinks" table above is **generated** from it by `bootstrap/gen-symlink-table.py`; run that after editing `dotfile_links` rather than hand-editing the table.
- **Per-host personal values** (e.g. the Samba `samba_lan_subnet`) live in an **untracked** `bootstrap/host_vars/<inventory_host>.yml` (template: the committed `…example`), kept out of this public repo by `.gitignore`. Ansible auto-loads it; role defaults carry RFC 5737 placeholders so a checkout is runnable without it.
- **Roles** (`bootstrap/site.yml`, sliceable by `--tags`): `packages` (apt — and, after install, it disables iperf3's bundled server daemon: iperf3 ships as a client-side throughput diagnostic, but its `iperf3.service` would otherwise listen on tcp/5201, so the role keeps the unit disabled+stopped), `dotfiles` (symlink the manifest + render `templated_configs` into `$HOME`, no root), `fonts` (the Nerd Fonts the configs reference — `Lilex`/`BigBlueTermPlus` — fetched from the pinned nerd-fonts release into `~/.local/share/fonts`, no root), `tailscale` (installs the Tailscale mesh VPN from its own signed apt repo — the share's transport + general remote reach; `tailscale up` auth stays manual), `samba` (layer-(a) `/etc/samba/smb.conf` + `/srv/smbshare` share — its `hosts allow` adds the Tailscale CGNAT range only when `enable_tailscale` is also on, otherwise LAN-only), `claude_user` (the `claude` user + `devshare` group + shared tree + repo ACLs — the *plumbing*; identity is manual), `credentials` (gnome-keyring launcher-untangle for login auto-unlock). There's also **`localbin`** — pinned prebuilt binaries from GitHub releases into `~/.local/bin` (no root), data-driven via `localbin_binaries` (**bluetuith**, **hcloud** the Hetzner Cloud CLI, and **tofu**/OpenTofu for Cloudflare-DNS-as-code, and **cf-terraforming** to export existing Cloudflare config into `.tf` — none in apt); idempotent via a per-(name, version) marker, sha256-verified, arch-matched from `uname -m` (with an optional per-entry `arch_map` override for releases that name x86_64 as `amd64` — hcloud/opentofu — and a `bash_completion` flag) — **`gtk_theme`** (no root), which builds the **hestia GTK3 theme** by downloading pinned `adw-gtk3` (sha256-verified), recolouring its accent to `#ce0056`, and installing it as `~/.local/share/themes/hestia`+`hestia-dark` (idempotent via a `(version, accent)` marker; see the GTK / Adwaita section) — and **`sway_session`** (root), which deploys the tracked session launch chain — `start-sway` → `/usr/local/bin` (greetd runs it; its NVIDIA env + `--unsupported-gpu` are applied only when an NVIDIA GPU is live) and the greetd `config.toml` → `/etc/greetd` — see the Sway section. Each role is idempotent — re-run with `--check` to verify `changed=0`. **`packages` + `dotfiles` + `fonts` + `localbin` + `gtk_theme` + `sway_session` are core; samba/tailscale/claude_user/credentials are optional**, gated by `enable_samba`/`enable_tailscale`/`enable_claude_user`/`enable_credentials` (`group_vars/all.yml`, default `true`) via `when: … | bool` — the §6 feature-flag surface; set one false (host_vars or `-e enable_samba=false`) to skip its role. A disabled feature also **skips its apt packages**: `package_group_features` maps gated groups (`sharing: [samba]`, `credentials: [gnome-keyring, libsecret-tools]`, `office: [libreoffice]`) to their toggle, and `packages` drops a group whose toggle is off (`acl` stays base — `claude_user`/`claude-access` need it). `enable_libreoffice` is a **package-only** toggle (no role of its own — it just gates the `office` group) and **defaults `false`** (libreoffice is heavy/optional — vifm's office-doc opener). There's also an opt-in **`nvidia`** role (`enable_nvidia`, **default `false`**): enables Debian non-free + installs `nvidia-driver` + kernel headers — host-specific, builds a DKMS module, and needs a **reboot** (which stays manual); `setup.sh` detects a card (lspci / sysfs vendor `0x10de`) and pre-answers the prompt.
- **Deliberately NOT Ansible-managed** (interactive, secret-handling, vendor, per-user, or heavy/optional — all in the runbook §8): NVM/Node, vim-plug + `:PlugInstall`, Claude Code itself; the `smbpasswd` Windows password; **`tailscale up`** (the Tailscale node auth — interactive SSO/auth-key; the *package* is now installed by the `tailscale` role); claude's identity keys + GitHub bot; the keyring/passphrase storage for credential auto-unlock. (Nerd Fonts *used* to be here — now the `fonts` role; the **bluetuith** binary too — now `localbin`; the **NVIDIA driver** too — now the opt-in `nvidia` role, though its **reboot** stays manual; the **Tailscale package** too — now the `tailscale` role, though `tailscale up` stays manual.)
- **Personalization seam** (`docs/personalizing.md`): three layers for making hestia someone else's system without editing role/playbook logic — (1) override values/lists in **host_vars** (toggles, plus wholesale `apt_packages:`/`nerd_fonts:`/`localbin_binaries:` overrides), (2) install anything custom via an untracked **`bootstrap/local.yml`** Ansible *tasks* file (gitignored; committed `local.yml.example`; `site.yml`'s final play `include_tasks` it **last** if present, `tag: local`; `become:` tasks reuse setup.sh's sudo password), (3) fork the `user/` configs. Goal 1 stays the tracked default; nobody edits `bootstrap/roles/` or `site.yml` to personalize. (Chosen over per-role `extra_*` vars, which don't scale.)
- **Samba share** (`system/samba/smb.conf.j2`, `docs/file-sharing-design.md`): SMB to a Windows laptop, POSIX-ACL permission model. The `samba` role **renders** the template (LAN subnet from host_vars) → `/etc/samba/smb.conf`. smbd listens on all interfaces; access is confined at the SMB layer by `hosts allow/deny` (loopback + home LAN, **+ the Tailscale CGNAT range when `enable_tailscale` is on** — the template gates that line and its Tailscale-flavoured comments on the toggle, so a Tailscale-less spin renders a clean LAN-only config) + SMB3 floor + a `nologin` `smbshare` principal.

## Tool configurations

### Vim (`user/vim/.vimrc`)
Uses **vim-plug** for plugins. To install plug.vim on a fresh system:
```sh
curl -fLo ~/.vim/autoload/plug.vim --create-dirs \
     https://raw.githubusercontent.com/junegunn/vim-plug/master/plug.vim
```
Then open Vim and run `:PlugInstall`. CoC extensions install automatically via `g:coc_global_extensions`. The config targets both Vim and Neovim — some plugins are conditional on `has('nvim')`.

Key plugins: `coc.nvim` (LSP/completion), `ctrlp.vim` (fuzzy finder), `vim-fugitive` (git), `copilot.vim` (requires Vim ≥ 9.0.0185), NERDTree. Theme: `wildcharm` (dark).

**Markdown viewing** — three layers:
- `:Glow` (or `<leader>md` = `\md`, markdown buffers only) renders the current file in **glow** as a full-screen pager in the **same terminal** (suspends Vim until `q`, like vifm), via `:!glow -p %`. Works in **both Vim and Neovim** — this is the plain-Vim path. Calls `GlowPreview()`; glow gets a real TTY so theming comes from `~/.config/glow/glow.yml` (see Glow section).
- `render-markdown.nvim` — live **in-buffer** rendering, **Neovim only** (needs treesitter/Lua, absent in plain Vim). Set up in a `has('nvim')` Lua block; headings themed to `#ce0056`. Dormant until `:PlugInstall`.
- `markdown-preview.nvim` — browser-tab live preview, **Neovim only**; kept as the escape hatch for diagrams/math.

### Neovim (`user/nvim/init.vim`, `user/nvim/lua/trees.lua`)
Neovim **shares the Vim config** rather than duplicating it. `init.vim` prepends `~/.vim` to `runtimepath` (so `plug.vim` in `~/.vim/autoload` and plugins in `~/.vim/plugged` are reused) and then `source`s `~/.vimrc`. The `has('nvim')` branches in `.vimrc` then activate the Neovim-only plugins (`render-markdown.nvim`, `markdown-preview.nvim`) and `lua require('trees')` (treesitter config in `user/nvim/lua/trees.lua`).

Fresh-install steps (Debian trixie ships **0.10.4**, recent enough — `apt install neovim`, package is `neovim`, binary is `nvim`):

### Bash (`user/bash/.bashrc`)
- Vi mode (`set -o vi`); git-aware prompt (cwd, branch, staged/unstaged/untracked counts, exit-status colour). The prompt is **identity-adaptive** from one shared rc: a normal login user gets `path branch $` (no username), but **root** prepends a **red** `\u` and the **`claude`** agent an **orange** (`38;5;208`) `\u` → `user path branch $`. Same file, coloured by `$EUID`/`$USER` — so `sudo su` shows red because **root sources this same `.bashrc`**, not the caller's (a "go red when root" test in the *caller's* prompt never fires once you're elevated). Needs the rc deployed where it's sourced: it's symlinked into dimitrios's home today; `claude`'s home gets it via a stopgap symlink (the bootstrap will own this once claude's env is automated — `tmp.txt`), and `/root/.bashrc` only goes red once the rc reaches root (a manual/optional `ln -s` — root's home isn't managed yet).
- Loads **NVM** on startup. Sets `GPG_TTY=$(tty)` (needed for terminal GPG signing), exports `EDITOR`/`VISUAL=vim`, and prepends `~/.local/bin` to `PATH` (guarded against duplicate entries). (The system-wide `editor` **alternative** is also pointed at `vim.basic` by the `packages` role — Debian defaults it to nano; git uses `core.editor=vim` regardless.)
- **`gpg-unlock`** function: caches the GPG signing passphrase for the session (prompts once via pinentry-tty), so a cold cache stops failing harness commits. Reads the key from git config; refuses to run under `$CLAUDECODE`. See the signing section.
- **SSH agent:** exports `SSH_AUTH_SOCK="$XDG_RUNTIME_DIR/openssh_agent"`. The agent itself is **Debian's shipped socket-activated user unit** (`/usr/lib/systemd/user/ssh-agent.socket`, already `enable`d — auto-starts on login); we deliberately use that rather than a custom unit. greetd doesn't source `.bashrc`, so this export covers interactive shells (terminals, Ansible). The SSH key is now **auto-loaded at login** by the credential-unlock hook (see the signing section), reading the passphrase from the gnome-keyring login keyring — so no per-boot `ssh-add`. It persists the whole session. (Fallback if the hook ever doesn't run: `ssh-add ~/.ssh/<your key>`, i.e. the `ssh_key_file` host_var.)
- **ssh TERM fix:** under kitty (`TERM=xterm-kitty`), `ssh` is aliased to advertise `TERM=xterm-256color` — servers rarely have the `xterm-kitty` terminfo, so without this backspace/arrows garble on remote shells. Plain `ssh` keeps working keys **and** the server's login MOTD; use `kitty +kitten ssh <host>` explicitly for full kitty integration remotely (it installs the terminfo but bootstraps via a command, so it skips the MOTD).
- Sources **`~/.bash_secrets`** for secrets (see above).
- No Rust: a previous `curl … | sh` rustup auto-installer block was removed (it could silently reinstall on shell startup). Install toolchains explicitly, never from `.bashrc`.

### Git (`user/git/`)
Default branch `main`, editor `vim`, commits GPG-signed. Global excludes at `~/.gitignore_global` (`user/git/.gitignore_global`, symlinked): the Claude `settings.local.json` and common macOS/Node/Vim/VS Code artefacts. It deliberately does **not** exclude `.gitignore`/`.editorconfig` — those are meant to be committed and shared per-repo, and a global exclude only applies on this machine so it wouldn't travel to clones/collaborators/CI anyway. Rule of thumb: only *never-commit-anywhere* noise goes in the global file; project ignores belong in a committed `.gitignore` (e.g. this repo's, protecting `bootstrap/host_vars/*.yml`); "just this clone" ignores go in `.git/info/exclude`. `core.excludesfile` uses git-native `~` expansion (`~/.gitignore_global`), and the gitconfig is **rendered** (`user/git/.gitconfig.j2`, not symlinked) so `gpg.program` templates from `{{ repo_root }}` — neither needs a manual edit on a home/repo move (`docs/repo-structure-design.md` §5). No HTTPS credential helper is configured — auth is over SSH (`git@github.com`), so an HTTPS remote would simply prompt; add `helper = cache` for an in-memory, no-plaintext fallback if ever needed (avoid `helper = store`, which writes plaintext to `~/.git-credentials`). The local `file://` transport is left at Git's CVE-2022-39253-safe default (`protocol.file = user`); if a local-submodule operation ever needs it, scope it per-command with `git -c protocol.file.allow=always …` rather than enabling it globally.

### Sway (`user/sway/config`)
Wayland compositor config. Notable points:
- **Keyboard** (`input type:keyboard`): `xkb_layout "us,de,gr,bg"` + `xkb_variant ",,,phonetic"` — Bulgarian uses the **traditional phonetic** variant (W → в); commas align variants to layouts. `xkb_options caps:super,grp:alt_shift_toggle` — Caps Lock = Super, **Alt+Shift** cycles layouts.
- **Gaps:** `gaps inner 8` with `gaps top -8` to cancel the gutter under Waybar so tiled windows sit flush (other edges keep the gap). `swaymsg reload` updates gap *defaults* but does **not** re-apply to existing workspaces — apply live with `swaymsg -- gaps top all set -8` (note the `--`, or swaymsg reads `-8` as a flag), or restart sway.
- **XWayland:** `xwayland enable` (needs the `xwayland` package). Changing it requires a **full sway restart**, not a reload.
- `for_window [app_id="floatterm"] floating enable, resize set 800 600` makes Waybar widget popups (`kitty --class floatterm -e <tui>`) open floating. The same rule backs the **`$mod+c` calculator** keybind (`kitty --class floatterm -e qalc` — **qalc**/Qalculate, the terminal calculator in the apt manifest; it replaced the accidentally-installed GTK4 gnome-calculator).
- **Screenshots — local vs shared (`user/sway/screenshot.sh`):** both `Print` and `$mod+Print` take a **region** screenshot (`grim -g "$(slurp)"`) via `screenshot.sh`; they differ only in destination. **`Print` → `~/Pictures/Screenshots`** (local, never exposed to claude). **`$mod+Print` → `/srv/clipshare`** (the *shared screenshot drop* — root:devshare, setgid + default ACLs, provisioned by the `claude_user` role next to `/srv/devshare`), which the **`claude`** user can `Read` (it's a `devshare` member; the Read tool ingests images). This is the supported way to show Claude Code an image: claude runs as a **separate user, walled off from this session's Wayland clipboard socket** (`/run/user/<dimitrios>`, `0700`), so a `Ctrl+V` image paste into Claude Code **can't work** — and opening that socket would expose every copied image/password. Sharing stays **explicit** (only `$mod+Print` shots are claude-visible) — take one, then tell claude "look at the screenshot". `shared` **falls back to the local dir** when `/srv/clipshare` is absent (a spin without `claude_user`), so the binding always works; the `notify-send` shows where the file landed. (The script also `mkdir -p`s the dir — fixing a latent bug where the old inline `Print` binding wrote into a possibly-missing `~/Pictures/Screenshots`.) Symlinked into `~/.config/sway/scripts/` like `start-waybar.sh`.
- **Screen-sharing env export:** `exec dbus-update-activation-environment --systemd WAYLAND_DISPLAY XDG_CURRENT_DESKTOP XDG_SESSION_TYPE` pushes the session env into the D-Bus/systemd user activation environment so `xdg-desktop-portal` activates with the right context. Required for screen sharing — see the **Screen sharing** section. `exec` (not `exec_always`) runs once at login; it does **not** re-run on `swaymsg reload`, so apply live after editing with the same command.
- **Session-script `exec` paths are path-generalised:** `exec …/credential-unlock.sh` and `exec_always …/start-waybar.sh` reference **`$HOME/.config/sway/scripts/…`** (sway runs `exec` via `sh -c`, so `$HOME` expands) — no clone-absolute path. `start-waybar.sh` is **symlinked** there; **`credential-unlock.sh` + its sibling `keyring-ssh-askpass.sh` are *rendered* there** (they carry identity values — `ssh_key_file` / `gpg_keygrip` / `git_signingkey` from host_vars), both into the same dir so `credential-unlock.sh` finds the sibling via `readlink -f "$0"`.
- **Launcher / NVIDIA workarounds (`system/sway-session/start-sway`):** sway isn't started by the config above — greetd runs **`/usr/local/bin/start-sway`** (`tuigreet … --cmd start-sway`), which sets the Wayland session env then `exec sway`. The NVIDIA-specific env (`GBM_BACKEND=nvidia-drm`, `__GLX_VENDOR_LIBRARY_NAME=nvidia`, `WLR_NO_HARDWARE_CURSORS=1`, `NVIDIA_DRIVER_CAPABILITIES=all`) + `--unsupported-gpu` are applied **only when an NVIDIA GPU is live** (guarded on `/dev/nvidia0` / the `nvidia` entry in `/proc/modules`) — on AMD/Intel they'd break rendering, so they're skipped. Both `start-sway` and the **greetd `config.toml`** that calls it are tracked (`system/sway-session/`) + deployed by the **`sway_session`** role (greetd config → `/etc/greetd/`, applied on next login — the role doesn't restart greetd). Only installing greetd/tuigreet themselves is a prereq (`system/sway-session/README.md`).

- **Idle inhibition (the "caffeine" replacement):** the auto-lock chain is `swayidle` (**pre-lock dim warning at 270s** → lock at 300s → DPMS-off at 600s) → `swaylock` (now reads `~/.config/swaylock/config`, wildcharm-themed — see the **Swaylock** section; the invocations carry no `-c`, so the config's `color` is the lock background). The **270s warning** dims *every* output to 40% via a **backgrounded `gammastep -O 6500 -b 0.4 &`** software-gamma drop (killed by `pkill -x gammastep` on the timeout's `resume`) **and** raises a persistent `notify-send -u critical -t 30000` notice (matches mako's `[urgency=high]` red style, stays the full 30s — previously it used the 5s default-timeout, so the warning vanished 25s before the lock). **gammastep must be backgrounded:** on Wayland its `-O` "one-shot" doesn't actually exit (wlroots resets the gamma ramp the instant the setting client disconnects, so gammastep stays alive to *hold* the dim) — and `swayidle -w` waits for each timeout command to finish, so a foreground gammastep would wedge swayidle forever (no resume fired, no lock). Backgrounding lets `notify-send` return; `resume` then `pkill`s gammastep (the disconnect restores brightness) **and dismisses the warning notice** — the `-t 30000` would otherwise leave it on screen for the full 30s after input already cancelled the lock, so `notify-send -p` stashes the notification id in `$XDG_RUNTIME_DIR/lock-warn.id` and `resume` runs `makoctl dismiss -n <id>` to clear that one (not `-a`, which would drop unrelated notifications on every wake). The dim is the real cue: this is a **desktop with external DisplayPort monitors and no `/sys/class/backlight`**, so `brightnessctl` can't dim the panels (the `XF86MonBrightness` keybinds are no-ops on this hardware) — software gamma dimming hits both 32" screens at once regardless of which has focus. Any input before 300s fires the resume (full brightness back, no lock). gammastep is in the apt manifest's `wayland` group. To stop it locking during fullscreen video, sway's native `inhibit_idle fullscreen` is set per media app — `for_window [app_id="firefox"|"chromium"|"mpv"|"^FFPWA-"] inhibit_idle fullscreen` — so those windows raise a Wayland idle inhibitor *while fullscreen* (swayidle honours it; the timers pause, then resume on un-fullscreen). Deliberately **scoped to media apps**, not `[app_id=".*"]`: a blanket rule would let a forgotten fullscreen terminal skip the auto-lock (the security boundary). `app_id` is unanchored regex, so `firefox` also matches `firefox-esr`, and **`^FFPWA-` matches every `firefoxpwa` web app** (Netflix, YouTube, …) — each PWA gets its own `FFPWA-<ULID>` app_id (not `firefox`), so the prefix match is what actually catches the common "fullscreen Netflix in a PWA" case; add a browser = one more line. (mpv also self-inhibits via `--stop-screensaver` by default; the sway rule is the belt-and-suspenders that doesn't depend on the app speaking the `idle-inhibit` protocol.)

Apply most config changes with `swaymsg reload`.

### Swaylock (lockscreen) (`user/swaylock/config`)
The lockscreen, on the unified theme (the **first app themed through `docs/theming.md`** — colours from `themes/wildcharm/palette.yml`). Sway invokes it as plain `swaylock -f` (the `-c 000000` was **dropped** from all three call sites — the keybind, the swayidle `timeout 300`, and `before-sleep`), so `~/.config/swaylock/config` is the single source for the lock colours — its `color=0a0a0a` is the background (previously pure black). ### Swaynag (dialogs) (`user/swaynag/config`)
The warning/error popup sway uses for confirmations — notably the `$mod+Shift+e` "exit sway?" dialog. Themed through `docs/theming.md` (palette colours as **bare `RRGGBB`**, no `#`). The **global** options set the dark base — `background=0a0a0a`, `border=1e1e1e`, `text=e0e0e0`, `button-background=111111`, `details-background=111111`, `accent` (`ce0056`) bottom border — for any dialog launched with **no `-t`**. **Gotcha:** swaynag's two built-in **types** (`warning`/`error`) ship *hardcoded* colours (warning = yellow bg), and the global options are **not** layered on top of them — only keys set *inside* that type's `[section]` override its presets. ### Screen sharing / xdg-desktop-portal (`user/xdg-desktop-portal/sway-portals.conf`)
Wayland screen sharing (Teams/Edge, OBS, etc.) goes through **PipeWire** + **xdg-desktop-portal**. On Sway the screencast backend is **`xdg-desktop-portal-wlr`** (wlroots); other portal interfaces (file picker, settings) fall to **`xdg-desktop-portal-gtk`**.

- **`sway-portals.conf`** routes interfaces per backend (matched to this session via `XDG_CURRENT_DESKTOP=sway`): `ScreenCast`/`Screenshot` → `wlr`, everything else `default=gtk`. Without explicit routing the portal can misroute screencast to gtk (which can't capture on wlroots) and silently fail. If gtk is ever uninstalled, the `default` must change (wlr only implements ScreenCast/Screenshot — no file picker).
- **Session env:** the portal must inherit `WAYLAND_DISPLAY`/`XDG_CURRENT_DESKTOP` via the Sway `dbus-update-activation-environment` exec line (see Sway section). The portal is **D-Bus-activated on demand** (first screencast request), so the env must be exported *before* it first activates — a fresh login handles this.
- **Monitor-only capture:** the wlr backend reports `AvailableSourceTypes=1` (MONITOR only) — **per-window sharing is not supported** on Sway; share a whole output. Workaround: isolate the app on the second monitor and share that output. May change once the `wlr` portal adopts `ext-image-copy-capture` toplevel sources.
- **Dark mode (color-scheme):** the gtk backend exposes `org.freedesktop.appearance color-scheme` via the **Settings** portal, which Firefox and other portal-aware apps read. Force dark with `gsettings set org.gnome.desktop.interface color-scheme prefer-dark` (needs `dconf-cli` + `libglib2.0-bin`); without it the portal reports `0` (no preference) and apps default to **light**. This surfaced right after the portal was first installed — Firefox flipped to a light theme until the setting was applied. **Now set at login** by a sway `exec` (D-Bus/portals section), so it's reproducible — see the **GTK / Adwaita** section.

**External dependencies** (install on a fresh system):
- `pipewire`, `pipewire-pulse`, `wireplumber` (usually present; the audio stack).
- `xdg-desktop-portal` + `xdg-desktop-portal-wlr` (screencast) + `xdg-desktop-portal-gtk` (file picker/other portals).
- `dbus-bin` provides `dbus-update-activation-environment` (used by the Sway env-export exec line).

### Waybar (`user/waybar/config`, `user/waybar/style.css`, `user/waybar/scripts/`)
Top bar; files symlinked into `~/.config/waybar/`. Reload with `killall -SIGUSR2 waybar` (or `killall waybar; waybar &`).

**Single launcher + boot race:** waybar is started by sway's `exec_always` line running `start-waybar.sh` and must be the *sole* launcher. Two boot problems, two fixes: (1) a globally-enabled `waybar.service` user unit double-started it — disabled with `sudo systemctl --global disable waybar.service`; (2) even solo, waybar launched before the compositor could map its layer-shell surface, so the bar didn't render until a `$mod+Shift+c` reload. `user/sway/start-waybar.sh` fixes (2) — it kills any instance, settles, and waits for an active output before `exec waybar`. (The session is greetd-launched, not systemd-managed, so a `graphical-session.target`-ordered user service isn't an option without extra plumbing.)

- **Font:** the **bar** uses `BigBlueTermPlus Nerd Font Mono` at 16px (`style.css`, the `*` rule). Bitmap-derived — keep its sizes at integer pixel-grid values (16/24/32) for crispness. **Not** the `437` variant: that one is the literal DOS code-page-437 build, which has no Greek *alphabet* (only ~12 Greek/math *symbol* glyphs — α π Σ Ω δ ε σ τ …), so a non-Latin window title (`sway/window`) rendered its `ε` as a euro-like lunate epsilon and fell back mid-word for every letter CP437 lacks. `Plus` is the same bitmap typeface with full Greek (and Cyrillic etc.) plus the same Nerd Font icon glyphs, so the look is unchanged but titles render cleanly. **Tooltips** are restyled via `tooltip`/`tooltip label` rules: `Lilex Nerd Font Mono` 17px (scalable, cleaner than the bitmap bar font, also full Greek coverage), on the unified `#0a0a0a` bg with a 1px `#ce0056` border + padding (GTK tooltips are otherwise translucent/borderless).
- **Icon alignment:** module icons are wrapped in inline Pango markup `<span size='xx-large' rise='-3072'>…</span>`. The `rise` is a hand-tuned vertical offset (Pango units, 1/1024 pt); re-tune if the font changes.
- **PUA glyph caveat:** editing tools can silently strip Private-Use-Area (Nerd Font) glyphs, leaving an empty `<span></span>`. If an icon renders as blank space, insert the codepoint via a small Python write and verify, rather than re-typing it.
- **Modules** (right side): `custom/cpu`, `memory`, `custom/gpu` (all GPUs, see below), `power-profiles-daemon`, `pulseaudio`, `user/sway/language`, `network`, `bluetooth`, `custom/usb`, `custom/notifications`, `custom/weather`, `clock`, `custom/debian`. Right-side modules share styling via a grouped CSS selector — add new ones to that group.
- **Fixed-width numeric values** keep the bar from resizing as values change width (e.g. `0%`→`100%`): percentages are right-padded to **2 digits** — custom modules in the script (`printf '%2d%%'` in `cpu.sh`/`gpu.sh`), built-in modules via fmt spec in the format (`{volume:>2}`, `{signalStrength:>2}`). (2 not 3: it reserves up to `99%`; the rare `100%` adds one char — an acceptable jump.) Leading spaces render fine in the bar label.
- **Click handlers** open floating TUIs as toggles so a second click closes the popup. Two toggle styles: for a known process name, `pgrep -x NAME && pkill -x NAME || kitty --class floatterm -e NAME` (network/bluetooth/pulseaudio). For the script-launched ones whose final process name isn't predictable (gpu/weather), **do NOT use `pgrep -f SCRIPT`** — it self-matches the launcher's own `sh -c` cmdline (which contains SCRIPT), so the first click kills the launcher and nothing opens. Toggle via the **sway window tree** instead: `swaymsg -t get_tree | grep -q floatterm-NAME && swaymsg '[app_id="^floatterm-NAME$"] kill' || kitty --class floatterm-NAME -e …` (the tree query sees actual windows, not process cmdlines; the unique `floatterm-NAME` app_id still floats under the existing `[app_id="floatterm"]` rule — sway app_id criteria are unanchored regex). These sway-tree popups launch kitty with `-o confirm_os_window_close=0` so closing them (the WM kills the window while a program runs) doesn't pop kitty's confirm dialog — the `pgrep -x` toggles don't need it because they `pkill` the process first, so kitty exits cleanly.
  - `custom/cpu` / `memory` → both toggle the same `htop` (`pgrep -x htop`; `htop` is in the apt manifest).
  - `network` → `nmtui`; ethernet shows the icon only (device name/IP in the tooltip).
  - `bluetooth` → `bluetuith`; right-click toggles the radio via `rfkill`.
  - `pulseaudio` → left-click mutes (`wpctl`); **right-click** toggles **`pulsemixer`** (PipeWire-native TUI mixer — WirePlumber persists levels; in the apt manifest).
  - `power-profiles-daemon` → left-click cycles profiles natively.
  - `custom/gpu` → toggles a live GPU monitor (`gpu-watch.sh`, app_id `floatterm-gpu`) in a floatterm sized larger (sway rule `1100x700`) so `nvidia-smi`'s ~91-col output doesn't wrap.
  - `custom/usb` → mounted USB/removable storage. **Shown only while a removable device is attached** (hidden otherwise); text = count of mounted removable filesystems (empty when attached but none mounted — your cue to click and mount). Tooltip lists each (device, size, label, mountpoint). Detects removable disks via `lsblk` (hotplug/rm/`tran=usb`). **Click → a wofi picker to mount/unmount** a partition via `udisksctl` (no root — polkit lets the active session mount removable media; `udisks2` is in the apt manifest); `notify-send` reports the result, and a successful **mount opens `vifm` at the new mountpoint** (parsed from udisksctl's output, launched detached). (`usb.sh menu`; clicking again `pkill`s the open wofi picker — toggle — rather than stacking another.)
  - `custom/notifications` → a static bell (`nf-md-bell-outline`, `tooltip-format: Notifications`); click toggles `notifications.sh show` (app_id `floatterm-notify`), which formats **`makoctl history`** via python into a pager — one line per notification (`● summary · app`). This mako build prints a human-readable text block (not JSON) and **records no timestamp**, so none is shown (the viewer parses both the text and the older JSON shape). No DND toggle (mako has no DND mode defined).
  - `custom/weather` → toggles the full multi-day forecast (`weather.sh show` → `curl wttr.in | less -R`, app_id `floatterm-weather`) in a floatterm.
- **`custom/gpu`** is **one widget for all GPUs** (`user/waybar/scripts/gpu.sh`, symlinked; `exec` via `$HOME`). The bar shows the **MAX** utilisation across cards (not an average — maxing a discrete dGPU on a dGPU+iGPU box reads 100%, not 50%); the tooltip lists **every** card (name · util · temp · VRAM). Prints nothing when no GPU is present (module hides). **Multi-vendor:** **NVIDIA** (`nvidia-smi`, matched to the DRM card by PCI bus id), **AMD** (amdgpu **sysfs** — `gpu_busy_percent`/hwmon/`mem_info_vram_*`, no extra tool), **Intel** (sysfs, best-effort — i915 has no util counter, so name+temp only). The **`on-click`** runs the companion **`gpu-watch.sh`** (symlinked), which picks the best live monitor available — `nvidia-smi`/`amdgpu_top`/`radeontop`/`intel_gpu_top`, falling back to a no-extra-tool sysfs poll; sway-tree toggle (app_id `floatterm-gpu`).
- **`custom/debian`** is a decorative Debian swirl pinned at the far right (accent red), non-interactive.
- **`custom/weather`** (`user/waybar/scripts/weather.sh`, symlinked) shows a **monochrome `nf-md-weather-*` glyph** + temperature in the bar (the condition text is mapped to a Nerd Font weather glyph in `weather_glyph()`, emitted by codepoint via `/usr/bin/printf '\U…'` so there's no literal PUA in the script; it sits in the same `<span size='xx-large' rise='-3072'>` icon span as the other modules, so the temp aligns with no rise hack) and location/condition/feels-like/humidity/wind in the tooltip, from **wttr.in** (`curl`, **IP-geolocated** — no location configured). Refreshes every **30 min** (`interval` — wttr.in etiquette) and **caches** the last good reading to `$XDG_RUNTIME_DIR/waybar-weather.json`, falling back to it on a network failure (prints nothing → module hidden if there's no data at all). `weather.sh show` (the on-click) opens the full forecast (`curl wttr.in | less -RS` — `-S` chops long lines instead of wrapping the wide 3-day output; the `floatterm-weather` popup gets a larger sway size rule, and kitty is launched with `-o confirm_os_window_close=0` so the sway-tree toggle closes it without a "program still running" prompt).
- **`custom/cpu`** (`user/waybar/scripts/cpu.sh`, symlinked) shows overall usage% in the bar (icon in the config format) and, on hover, **CPU temperatures** + per-core usage (laid out as a monospace `<tt>` grid, 8 per row, so it doesn't wrap). Usage is a `/proc/stat` delta vs the previous sample cached in tmpfs (`$XDG_RUNTIME_DIR/waybar-cpu.prev` — no `sleep`; first tick reads 0%). Temps come from the `k10temp`/`coretemp`/`zenpower` hwmon via **sysfs** (no lm-sensors): labelled `Tctl`/`Tccd*` on AMD (per-CCD), `Core N` on Intel (per-core). Replaced the built-in `cpu` module (which can't surface temps).
- **Built-in tooltips:** `memory` hover shows used/total/free + swap (`tooltip-format` with `{avail}`/`{swap*}`); `clock` is compact — `{:%a %d, %H:%M:%S}` (e.g. `Mon 21, 14:30:45`; year/month live in the hover tooltip, since you rarely forget those, but seconds matter for maintenance windows). `interval: 1` is cheap (a pure strftime re-render, no subprocess). Hover shows the full date `%A, %d %B %Y  %Z` (with timezone) + a **native month calendar** (`{calendar}`); neither the date nor the calendar carries a `<small>` wrapper, so both render at the **full 17px tooltip base** — matching the other widgets' tooltips — waybar v0.12, no external `cal`. Keep the calendar inside `<tt>` (monospace grid alignment) — the calendar rendering is finicky, so change its wrapper cautiously. The `actions` block uses waybar's **built-in** clock actions (not shell commands): `on-scroll-up`/`-down` = `shift_up`/`shift_down` (flip month on hover-scroll), `on-click` = `shift_reset` (jump the calendar back to **today**).

**External dependencies** (install on a fresh system):
- `network`: NetworkManager (`nmtui`, `nmcli`) — usually present.
- `bluetooth`: `bluez` (apt) + `bluetuith` (not in apt; the **`localbin` role** fetches the pinned [bluetuith release](https://github.com/bluetuith-org/bluetuith/releases) binary into `~/.local/bin` — see *Bootstrap*). `systemctl enable --now bluetooth`.
- `power-profiles-daemon`: apt package; `systemctl enable --now power-profiles-daemon`. Works with the `amd-pstate-epp` cpufreq driver (power-saver/balanced/performance). The module hides itself if the daemon's dbus name is absent.
- `custom/gpu`: vendor-dependent — **NVIDIA** needs `nvidia-smi` (the opt-in `nvidia` role / `enable_nvidia`); **AMD** needs nothing (amdgpu sysfs); **Intel** shows name+temp from sysfs (full util needs `intel-gpu-tools`). The `on-click` monitor uses whatever live tool is present (`nvidia-smi`/`amdgpu_top`/`radeontop`/`intel_gpu_top`) or a sysfs poll, so no package is strictly required.
- `custom/cpu`: nothing beyond coreutils (`/proc/stat` + sysfs hwmon). `custom/weather`: `curl` (in the manifest) + the Nerd Font (for the weather glyph) + GNU coreutils `printf` (for `\U`); network for live data (caches otherwise). No emoji font needed anymore (glyph is monochrome Nerd Font).
- Fonts (`BigBlueTermPlus` / `Lilex` Nerd Font) are installed by the **`fonts` role** into `~/.local/share/fonts/` (from the pinned nerd-fonts release; bump `nerd_fonts_version` in `bootstrap/roles/fonts/defaults/main.yml` to update). Adding a new Nerd Font = a new `nerd_fonts` entry (with its `marker` ttf) + `fc-cache` (the role's handler).

### Kitty (`user/kitty/kitty.conf`, `user/kitty/music.session`)
Terminal. Font `Lilex Nerd Font Mono` at 11pt; `ctrl+shift+=/-/0` adjust size. **Color emoji** (e.g. `✅` in program output) need **`fonts-noto-color-emoji`** (apt) — Nerd Fonts don't carry the Unicode emoji set, so without it emoji render as tofu (double-width empty boxes). kitty reaches it via fontconfig fallback and is presentation-aware (text glyphs like `✓`/`✗` stay in the primary font), so **no `symbol_map`** is set — a broad one would wrongly force those text symbols into the emoji font. 16-colour palette from `wildcharm`; cursor/selection/active-tab/ANSI-red use accent `#ce0056`. `music.session` (bound to `$mod+m` in Sway) lays out **cava** as a short visualiser bar on top and **cmus** below (focused via the `focus` directive); the cmus launch is wrapped so quitting cmus also closes cava, tearing down the whole window.

### cmus (`user/cmus/rc`)
Startup commands sourced after cmus restores its saved options. wildcharm-matched 256-colour theme; accent tracks terminal colour `1` (`#ce0056`). The `rc` is **rendered** (`user/cmus/rc.j2`, not symlinked): the music-library path is host-specific data, so the browser default + the `U` rescan binding come from **`cmus_music_dir`** (default `~/Music`, overridden per host in `bootstrap/host_vars/<host>.yml`; `docs/repo-structure-design.md` §5). The library is **not** auto-synced: `U` re-scans that folder for newly added tracks (re-add, dedupes), `u` = `update-cache` (prune deleted / refresh changed tags).

### cava (`user/cava/config`)
Spectrum visualiser. PipeWire input (monitor of the default sink). 8-stop accent gradient in the `#ce0056` family, fine bars (`bar_width 1` / `bar_spacing 0`), `noise_reduction = 88` (int 0–100 scale) + `monstercat` smoothing. A `method = sdl` GPU window (smooth pixel bars + GLSL shaders) is available — cava here is built with SDL2 — if a fancier visualiser is ever wanted.

### mako (`user/mako/config`) / wofi (`user/wofi/config`, `user/wofi/style.css`)
Notification daemon and app launcher (`$mod+d`), both on the unified `#0a0a0a` / `#ce0056` theme. Reload mako with `makoctl reload`; wofi restyles on next launch. `max-history=100` (up from mako's default 5) so the waybar notification widget (`custom/notifications` → `makoctl history`) has a deeper buffer — still in-memory only (lost on a mako restart/reboot; mako has no on-disk persistence).

### GTK / Adwaita (`user/gtk/`, `bootstrap/roles/gtk_theme/`)
GTK theming for the unified look (`docs/theming.md`). The hard-won outcome after a deep investigation: **GTK3 apps get dark + the exact `#ce0056` accent; GTK4/libadwaita apps get dark only** (their accent is locked on Debian's libadwaita — see below). Pieces:
- **The hestia GTK3 theme = recoloured adw-gtk3.** libadwaita-matching GTK3 theme **`adw-gtk3`** (not in Debian apt) is downloaded pinned (`v6.5`, sha256-verified) by the **`gtk_theme` role**, which seds its accent `@define-color accent_bg_color @blue_3` → **`#ce0056`** and installs it as `~/.local/share/themes/hestia` + `hestia-dark`. No compiler: adw-gtk3 references `@accent_bg_color` at **runtime** (156×), so one `@define-color` edit recolours every accent surface (focus rings, selection, tabs, shaded hovers). Idempotent via a `(version, accent)` marker. This covers GTK3 apps — **Remmina, GIMP, Firefox's file chooser**.
- **Selected by `GTK_THEME`, with gsettings for the portal.** The decisive selector for in-process GTK3 apps (Remmina, GIMP) is **`export GTK_THEME=hestia-dark` in `start-sway`** — `GTK_THEME` is the highest-priority theme source and **overrides both gsettings `gtk-theme` and `~/.config/gtk-3.0/settings.ini`** (in a non-GNOME sway session those two turn out to be *ignored* for the theme name — a hard-learned gotcha: the launcher's old `GTK_THEME=Adwaita:dark` was silently forcing stock Adwaita and masking every gsettings/settings.ini/gtk.css change we made). Separately, a sway `exec` sets gsettings `color-scheme prefer-dark` (dark for portal-aware/Firefox/libadwaita) **and** `gtk-theme hestia-dark` (the **portal file chooser** reads gsettings, not `GTK_THEME`/settings.ini). `gtk-3.0/settings.ini` (also `hestia-dark`) + the `gtk-3.0/gtk.css` overlay are kept belt-and-suspenders.
- **GTK4 / libadwaita accent is a hard dead end on trixie.** Proven (three ways, cleanly isolated) that **libadwaita 1.7 locks the accent against user CSS** — neither `@define-color accent_bg_color` nor the 1.8-style `:root { --accent-bg-color }` recolours gnome-calculator (the file *loads* — a green-window control showed — but the accent doesn't budge). And libadwaita ignores external themes entirely, so no theme helps. `gtk-4.0/gtk.css` carries the `:root` override anyway — **inert now, auto-active when Debian reaches libadwaita 1.8+**. The portal `accent-color` is also moot (a nine-colour named enum, and `xdg-desktop-portal-gtk 1.15.3` doesn't even export it). So GTK4/libadwaita apps (gnome-calculator, nautilus) are **dark with the default blue accent** until 1.8.
- **Firefox & its `firefoxpwa` PWAs follow hestia-dark.** They're GTK3, so `GTK_THEME=hestia-dark` themes their **chrome, menus, and file chooser** to the red accent — no `userChrome` needed (the earlier "Firefox is out of reach" worry was just the `GTK_THEME=Adwaita:dark` bug masking it; each PWA is a `FFPWA-*` GTK3 window and picks it up too). Dark web *content* comes from `color-scheme`; the only conceivable holdout is in-page form controls drawn by Firefox's own engine, which `userContent.css` could address if ever wanted.
- **Deferred (gradual):** base/surface colours toward the exact hestia palette (`#0a0a0a`/`#111111`/…) in the theme, **Breeze** (the Qt/KDE companion), and the libadwaita accent (revisit at 1.8). Status table in `docs/theming.md`.
